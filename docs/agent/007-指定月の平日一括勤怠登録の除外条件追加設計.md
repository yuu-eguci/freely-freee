# 007

## 要望

006 で設計した「指定の月の平日に勤怠を詰め込む」へ、以下の除外条件を追加する。

- 現在の除外条件:
  - `day_pattern != "normal_day"` は処理対象から外す
- 追加したい除外条件:
  - `use_default_work_pattern == false` も処理対象から外す

## 目的

- 会社や本人の個別勤務パターンが適用されている日 (`use_default_work_pattern == false`) には、自動投入を行わない
- 既存の normal_day 判定と合わせて、意図しない上書きを防ぐ

## 対象範囲

- `app/actions/bulk_attendance.py` の日次処理判定
- 設計ノート 006 の判定条件説明・ログ仕様の更新
- 必要ならスキップ件数サマリーの定義更新

## 非対象

- 認証フロー
- タグ付与ロジックの仕様変更
- 勤務時刻 (`09:00-19:00`, `11:30-12:30`) の変更

## 詳細設計

### 1. 追加仕様の要点

006 の日次判定を以下に拡張する。実行対象日は肯定形で次のように定義する。

- 実行対象条件:
  - `day_pattern == "normal_day"`
  - `use_default_work_pattern == true`
- 上記のどちらかを満たさない日はスキップ

ねらいは「`normal_day` でも個別勤務パターン適用日には自動投入しない」こと。

### 2. 判定順序と判定条件

`GET /work_records/{date}` のレスポンスを使って、1 日ごとに次の順で判定する。

1. `day_pattern` 判定
   - 条件: `day_pattern == "normal_day"` であること
   - 判定NG時: スキップ (`reason_code=day_pattern_not_normal_day`)
2. `use_default_work_pattern` 判定
   - 条件: `use_default_work_pattern == true` であること
   - 判定NG時: スキップ (`reason_code=use_default_work_pattern_false`)
3. 上記 1, 2 を両方通過した日のみ、勤怠 PUT とタグ PUT を実行

実行条件 2 つ (`day_pattern == "normal_day"` / `use_default_work_pattern == true`) を同時に満たさない日は、
判定順序に従って `day_pattern_not_normal_day` を採用する。
このときスキップ理由は 1 件のみ記録し、重複カウントはしない。

判定順は 006 との整合を優先して `day_pattern` を先に評価する。
この順にすることで、休日・所定休日などは従来どおり最初の条件で早期スキップできる。

### 3. スキップ理由のログ仕様

既存の `[SKIP]` 形式を維持しつつ、理由コードを明示する。

- 出力形式:
  - `[SKIP] <date> reason=<reason_code> detail=<key>=<value>`
- 例:
  - `[SKIP] 2026-03-01 reason=day_pattern_not_normal_day detail=day_pattern=prescribed_holiday`
  - `[SKIP] 2026-03-10 reason=use_default_work_pattern_false detail=use_default_work_pattern=false`

サマリーは既存 (`登録日数 / スキップ日数 / エラー日数`) を維持し、必要ならスキップ内訳を追加できる形にする。

- 追加候補:
  - `skip_day_pattern_count`
  - `skip_default_pattern_false_count`

カウント方針は「1 日につきスキップ理由は 1 つ」。
よって `skip_day_pattern_count` と `skip_default_pattern_false_count` は相互排他的に加算し、重複加算しない。

`use_default_work_pattern` は `GET /work_records/{date}` レスポンスの boolean フィールドとして扱う。
ログ上の `detail=use_default_work_pattern=false` はこの boolean 値を文字列表現したもの。

### 4. 既存 006 との整合性

今回の追加は「スキップ条件の拡張」のみで、006 の主フローは維持する。

- 維持する項目
  - 実行順序 (`work_record` 取得 -> 判定 -> 勤怠 PUT -> タグ PUT)
  - エラー時即中断
  - 終了コードの扱い (`0 / 1 / 2 / 130`)
  - ログプレフィックス (`[OK] / [SKIP] / [ERR]`)
  - 0.5 秒待機の方針
- 更新する項目
  - 006 の「normal_day なら続行」の記述を「`day_pattern == "normal_day"` かつ `use_default_work_pattern == true` の場合に続行」へ更新
  - 006 のスキップログ例に `reason_code` を追加

006 側の更新タイミングは「007 の設計レビュー完了後、実装着手前」に確定する。
運用上は 007 の実装 PR と同一タイミングで 006 の記述更新を入れ、仕様と実装の差分を残さない。

### 5. `use_default_work_pattern` の確認元（参照先）

確認元は以下の 2 つとする。

- `docs/agent/006-指定月の平日一括勤怠登録設計.md` に記載済みの `GET /work_records/{date}` 実レスポンス例
- 同ドキュメント内の `PUT /work_records/{date}` 成功レスポンス例（`use_default_work_pattern: true` を確認）

この設計では、まず 006 設計ノートの実レスポンス記録を正として実装する。
将来 API 仕様変更があった場合は freee 公式 API ドキュメントで差分確認し、006/007 を同時更新する。

### 6. 実装時の変更対象ファイル案

今回は設計追記のみ。実装フェーズでの変更対象は以下を想定する。

- `app/actions/bulk_attendance.py`
  - `_process_date()` の実行条件を `day_pattern == "normal_day"` かつ `use_default_work_pattern == true` に更新
  - `[SKIP]` ログの理由コード出力を追加
  - 必要に応じてサマリーのスキップ内訳カウントを追加
- `docs/agent/006-指定月の平日一括勤怠登録設計.md`
  - 判定条件・ログ仕様の記述を今回仕様に合わせて更新
- `docs/agent/007-指定月の平日一括勤怠登録の除外条件追加設計.md`
  - 本ノートを実装確定内容で追記・確定化

## レビュー

### 1st review

- 判定順序 (`day_pattern` -> `use_default_work_pattern`) の方針は妥当。ただし両条件が同時に真になる日の扱い (どちらの理由でカウント・記録するか) を明記したほうがよい
- スキップログの `detail=use_default_work_pattern=false` は、レスポンスが boolean で返る前提を明文化しておくと実装時の解釈ブレを防げる
- サマリーの追加候補 (`skip_day_pattern_count`, `skip_default_pattern_false_count`) は、重複カウントを許すかどうかの方針を先に決めておくべき
- 「`use_default_work_pattern != false`」は否定の否定で読みづらいため、`use_default_work_pattern == true` へ統一したほうが明快
- 006 側の記述更新タイミング (007 実装前/実装時/実装後) を決めておくと運用が安定する
- `GET /work_records/{date}` の `use_default_work_pattern` フィールド存在・型について、確認元 (API ドキュメントか実レスポンス) の参照先を明記するとより安全

### 対応ログ

- 指摘1対応: 両条件同時不一致時は `day_pattern_not_normal_day` を優先し、1日1理由で記録する方針を詳細設計へ追記
- 指摘2対応: `use_default_work_pattern` を `GET /work_records/{date}` の boolean フィールドとして扱うことと、ログ値の表現方法を明記
- 指摘3対応: スキップ内訳カウントは相互排他で重複加算しない方針を明記
- 指摘4対応: 条件式を肯定形 (`day_pattern == "normal_day"` / `use_default_work_pattern == true`) に統一
- 指摘5対応: 006 更新タイミングを「007 設計レビュー完了後、実装着手前。実装 PR と同時更新」に確定
- 指摘6対応: `use_default_work_pattern` の参照先を 006 の実レスポンス記録として明記

判定: LGTM 依頼

- スキップ理由の相互排他カウント方針により、サマリー合計値との整合を保てる
- 条件式を肯定形に統一したことで、判定ロジックの可読性と保守性が向上している
- 006 更新タイミングの明記により、ノート間の仕様ズレを防ぐ運用になっている
- 参照先が整理されており、将来の API 仕様差分追跡も行いやすい
- 対象範囲/非対象範囲と変更対象ファイル案が具体的で、実装着手条件が揃っている
