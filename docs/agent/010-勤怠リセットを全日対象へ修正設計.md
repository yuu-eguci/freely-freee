# 010

## 要望

008 の設計・実装で「006 + 007 と同じ条件で処理対象を見つける」としたが、これは誤りだった。

- 修正したい内容:
  - 「指定月の平日勤怠をリセットする」機能は、テスト用リセットが目的なので **指定月の全日を対象** にする
- つまり、次の判定で除外しない:
  - `day_pattern == "normal_day"`
  - `use_default_work_pattern == true`

## 目的

- テスト前の初期化として、指定月の日次状態を一括で確実に空に戻す
- 条件判定に起因する取りこぼしをなくす

## 対象範囲

- 008 で定義した「指定月の平日勤怠をリセットする」設計の修正
- 全日対象へ変更するための処理フロー・ログ・サマリー仕様更新
- 実装時に影響する変更対象ファイルの特定

## 非対象

- 006/007 の「勤怠を詰め込む」機能の対象日判定仕様変更
- 認証フロー・トークン管理の変更
- 今回タスクでのコード実装

## 詳細設計

### 1. 設計ゴール

指定月 (`yyyy-mm`) の 1 日から末日までを **全日対象** として処理し、各日付に対して以下 2 API を順に実行して勤怠登録状態を初期化する。

- `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}` に `break_records: []`, `work_record_segments: []`
- `PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}` に `employee_attendance_tags: []`

`day_pattern` / `use_default_work_pattern` は実行可否判定に使わない。

### 2. 008 との差分（削除/維持）

#### 削除するもの

- 006+007 と同じ対象日判定を使う前提
- `GET /work_records` 取得結果を用いた以下のスキップ判定
  - `day_pattern != normal_day`
  - `use_default_work_pattern != true`
- 上記判定に紐づく `SKIP` 理由コード出力
  - `day_pattern_not_normal_day`
  - `use_default_work_pattern_false`

#### 維持するもの

- メニュー名: `指定月の平日勤怠をリセットする`（既存文言を維持）
- 対象月入力 (`yyyy-mm`) と月内日付ループ
- `GET /hr/api/v1/users/me` で `company_id`, `employee_id` を解決
- リセット対象 API と payload
  - `work_records`: 空配列で上書き
  - `attendance_tags`: 空配列で上書き
- エラー時即中断（ロールバックなし）方針
- API 呼び出し間の待機定数方針（`BULK_ATTENDANCE_API_WAIT_SECONDS = 0.5`）

### 3. 処理フロー（全日対象版）

```text
1. メニューから「指定月の平日勤怠をリセットする」を選択
2. input() で対象月 yyyy-mm を受け取り、妥当性を検証
3. GET /hr/api/v1/users/me で company_id, employee_id を取得
4. 対象月の日付リスト (1日〜末日) を生成
5. 日付ループ（全日対象）
   5-1. PUT /hr/api/v1/employees/{employee_id}/work_records/{date}
        body: { company_id, break_records: [], work_record_segments: [] }
   5-2. PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}
        body: { company_id, employee_attendance_tags: [] }
   5-3. 日次ログ出力 (OK)
   5-4. 共通定数秒数だけ待機
6. エラーがなければ完了サマリー出力
7. 途中エラー時はサマリーを出して即中断
```

### 4. ログ仕様（SKIP の扱い変化）

#### OK

- 形式: `[OK]   <date> 勤怠/タグを空へ更新済み`
- 例: `[OK]   2026-03-10 勤怠/タグを空へ更新済み`

#### SKIP

- 本修正後は、日次対象判定を廃止するため **原則出力しない**
- `skip_count` は常に `0`（異常系の明示スキップ要件を追加しない限り増加しない）
- 既存実装に `SKIP` 出力分岐が残っている場合は、全日対象仕様との齟齬になるため削除対象とする

#### ERR

- 形式: `[ERR]  <date> <prefix>: <api_error_messages>`
- 例: `[ERR]  2026-03-15 勤怠タグリセット失敗: 給与明細が確定されているので、更新できません。`

`ApiResponseError.response_body.errors[].messages` が取得できる場合は結合表示し、取得できない場合は例外文字列を表示する。

### 5. サマリー仕様

- 正常完了:
  - `完了: <ok>日成功 / 0日スキップ / 0日エラー`
- 途中中断:
  - `中断: <ok>日成功 / 0日スキップ / 1日エラー (<date> で中断)`

補足: 008 時点で `skip_count` を持つ実装構造でも、値は常に `0` になる。将来的な誤解を避けるため、実装では `skip_count` 自体の削除も検討対象とする。

### 6. 変更対象ファイル案

今回は設計のみで、以下は実装時の変更候補。

- 変更案: `app/actions/bulk_attendance_reset.py`
  - 判定 GET (`work_records`) と `SKIP` 分岐を削除
  - 全日を `work_records` / `attendance_tags` の 2 PUT で処理
- 変更案: `app/actions/bulk_attendance_common.py`
  - `evaluate_target_day(...)` をリセット機能側から切り離し
  - `print_summary(...)` の `skip_count` 依存を見直し（残すなら常に 0、削除するなら関数シグネチャ更新）
- 変更案: `docs/agent/008-指定月の平日一括勤怠リセット設計.md`
  - 誤った対象日判定前提を廃止し、010 への参照または差分注記を追記
- 変更案: `docs/agent/010-勤怠リセットを全日対象へ修正設計.md`
  - 本書（確定版）を正として維持

### 7. リスクと確認観点

#### リスク

- 全日対象化により API 呼び出し回数が増え、月末日数分の PUT を必ず実行する
- `skip_count` の表示/内部カウントが残存した場合、仕様理解のノイズになる
- 既存テストやログ期待値が `SKIP` を前提にしている場合は回帰が起こる

#### 確認観点

- 指定月の土日祝相当日でも `OK` または `ERR` となり、`SKIP` が出ないこと
- `day_pattern` / `use_default_work_pattern` の値に関係なく処理対象になること
- サマリーのスキップ件数が `0` で一貫すること
- `work_records` 成功後に `attendance_tags` が失敗した際、即中断・ロールバックなしが維持されること
- 待機定数（0.5秒）が従来どおり適用され、過度な連続リクエストにならないこと

- 削除した仕様:

  - 上記に紐づく `SKIP` 理由コード出力

  - 日次は `OK/ERR` 中心、`SKIP` は原則廃止
  - サマリーのスキップ件数は `0` 固定
- 実装反映ファイル:
  - （例）`app/actions/bulk_attendance_reset.py`
  - （例）`app/actions/bulk_attendance_common.py`
  - （必要時）関連テスト、設計ドキュメント

  - エラー時中断挙動が想定どおりであること
