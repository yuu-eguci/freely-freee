## 要望

- つまり、次の判定で除外しない:

  - `use_default_work_pattern == true`

## 目的

## 詳細設計

### 1. 設計ゴールと全日対象の妥当性根拠

指定月 (`yyyy-mm`) の 1 日から末日までを **全日対象** として処理し、各日付に対して以下 2 API を順に実行して勤怠登録状態を初期化する。

- `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}` に `break_records: []`, `work_record_segments: []`
- `PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}` に `employee_attendance_tags: []`

全日対象とする根拠は次のとおり。

- 本機能は「テスト前に月全体を初期化する」用途であり、営業日判定より取りこぼし防止を優先する
- freee 上では休日相当日にも過去投入データ・手動編集・他検証の残骸が存在しうるため、`day_pattern` ベースの絞り込みは初期化漏れの原因になる
- リセット payload は同一内容の再 PUT で収束するため、全日一括実行の運用と相性がよい

### 2. メニュー文言の確定方針

機能実態に合わせ、メニュー表示名は以下に変更する。

- 変更後: `指定月の勤怠をリセットする`

旧文言 `指定月の平日勤怠をリセットする` は誤解を生むため採用しない。

### 3. API 方針と `GET /work_records` の扱い確定

#### 利用する endpoint（確定）

- `GET /hr/api/v1/users/me`
- `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}`
- `PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}`

#### `company_id` 受け渡し方針（確定）

- `GET /hr/api/v1/users/me`: query 不要
- PUT (`work_records`, `attendance_tags`): body に `company_id` を付与
- PUT で query の `company_id` は付与しない

#### `GET /work_records/{date}` の扱い（確定）

- 全日対象化後は、対象日判定用途の `GET /work_records/{date}` は **実行しない（削除）**
- これにより 008 の `day_pattern` / `use_default_work_pattern` 判定系はリセット機能から廃止する

### 4. 処理フロー（全日対象版）

```text
1. メニューから「指定月の勤怠をリセットする」を選択
2. input() で対象月 yyyy-mm を受け取り、妥当性を検証
3. GET /hr/api/v1/users/me で company_id, employee_id を取得
4. 対象月の日付リスト (1日〜末日) を生成
5. 日付ループ（全日対象）
   5-1. PUT /hr/api/v1/employees/{employee_id}/work_records/{date}
        body: { company_id, break_records: [], work_record_segments: [] }
   5-2. PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}
        body: { company_id, employee_attendance_tags: [] }
   5-3. 日次ログ出力 (OK)
   5-4. 共通定数秒数だけ待機
6. エラーがなければ完了サマリー出力
7. 途中エラー時はサマリーを出して即中断
```

### 5. ログ/サマリー仕様（`skip_count` の扱い確定）

#### 方針

- `SKIP` ログは廃止
- `skip_count` はリセット機能から削除（変数・サマリー表示ともに削除）
- リセット機能のサマリーは `ok_count` / `error_count` のみを扱う

#### OK

- 形式: `[OK]   <date> 勤怠/タグを空へ更新済み`

#### ERR

- 形式: `[ERR]  <date> <prefix>: <api_error_messages>`

#### サマリー

- 正常完了:
  - `完了: <ok>日成功 / 0日エラー`
- 途中中断:
  - `中断: <ok>日成功 / 1日エラー (<date> で中断)`

補足: 006/007 の `skip_count` 仕様は維持し、本リセット機能でのみ削除する。

### 6. レート制限観点（呼び出し回数見積と待機方針）

#### 呼び出し回数見積

- 月内日数: 28〜31 日
- 日次 API: 1 日あたり 2 回 PUT
- 月次合計 PUT: `56〜62` 回
- 初期 `GET /users/me` を含む総呼び出し: `57〜63` 回

#### 待機秒数方針

- `BULK_ATTENDANCE_API_WAIT_SECONDS = 0.5` を維持
- 理由:
  - 逐次実行でバースト送信を避ける
  - 月最大 62 PUT でも最小待機時間は約 31 秒となり、短時間集中を緩和できる
- 追加運用:
  - `429` 系エラー発生時は当日で中断し、再実行時に同値 PUT で収束させる
  - 必要に応じて待機秒数を設定値化して調整可能にする（実装時判断）

### 7. エラー時中断とリカバリー手順

#### 中断方針

- `work_records` / `attendance_tags` のいずれかで失敗した時点で即中断
- ロールバックは実施しない（跨りトランザクション非対応のため）

#### 途中失敗時の状態

- `work_records` 成功後に `attendance_tags` 失敗した場合、当日は「勤怠のみ空・タグ未更新」が残る可能性がある

#### リカバリー手順（運用確定）

1. `[ERR]` ログから失敗日と API メッセージを特定する
2. 原因を解消する（例: 対象月の確定解除、権限不足の是正、一時的 API エラー解消）
3. 同じ `yyyy-mm` で本機能を再実行する
4. 再実行後サマリーが `完了` になることを確認する
5. 必要時のみ失敗日をスポット確認する（`work_records` / `attendance_tags` の状態確認）

### 8. 008 との差分（削除/維持）

#### 削除するもの

- 006+007 と同じ対象日判定を使う前提
- `GET /work_records` 取得結果を用いたスキップ判定
  - `day_pattern != normal_day`
  - `use_default_work_pattern != true`
- `SKIP` 理由コード出力
  - `day_pattern_not_normal_day`
  - `use_default_work_pattern_false`
- リセット機能における `skip_count`

#### 維持するもの

- 対象月入力 (`yyyy-mm`) と月内日付ループ
- `GET /hr/api/v1/users/me` で `company_id`, `employee_id` を解決
- リセット対象 API と payload
  - `work_records`: 空配列で上書き
  - `attendance_tags`: 空配列で上書き
- エラー時即中断（ロールバックなし）
- API 呼び出し間の待機定数方針（`BULK_ATTENDANCE_API_WAIT_SECONDS = 0.5`）

### 10. リスクと確認観点

#### リスク

- 全日対象化により月内全日分の API 呼び出しを必ず実行する
- 途中失敗時に当日の `work_records` / `attendance_tags` が非対称状態で残る可能性がある
- メニュー文言を変更しない実装が混入すると仕様誤認を招く

#### 確認観点

- メニュー表示が `指定月の勤怠をリセットする` になっていること
- 土日祝相当日でも `OK` または `ERR` となり `SKIP` が出ないこと
- 月最大ケース（31 日）でも呼び出し回数見積どおりで、待機秒数が適用されること
- サマリーが `ok/error` のみで表示されること
- 途中失敗後に同月再実行で空状態へ収束すること
