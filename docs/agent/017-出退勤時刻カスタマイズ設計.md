# 017 出退勤時刻カスタマイズ設計

## 要望

- "指定の月に自分の勤怠を詰め込む" 機能は、現在 09:00 ~ 19:00 固定で勤怠を自動記録する
- そこで、以下のような入力を設けたい
    - "出勤何時にする？ ( そのまま Enter 押せば 9 時になる )" 数値入力
    - "退勤何時にする？ ( そのまま Enter 押せば 19 時になる )" 数値入力

## 詳細設計

### 1. 現在のハードコード箇所の特定

- `app/actions/bulk_attendance_common.py`
    - `BULK_ATTENDANCE_WORK_START_MINUTES = 9 * 60`
    - `BULK_ATTENDANCE_WORK_END_MINUTES = 19 * 60`
    - `BULK_ATTENDANCE_BREAK_START_TIME = "11:30:00"`
    - `BULK_ATTENDANCE_BREAK_END_TIME = "12:30:00"`
- `app/actions/bulk_attendance.py`
    - `_build_default_work_record_payload` で上記定数を `work_record_segments` と `break_records` に直結している
    - `_build_half_decision` で半休判定の境界を `09:00-19:00` 前提で比較している
    - `_work_result_label` で表示ラベルを `09:00-19:00` 前提で組み立てている
- `app/actions/bulk_attendance_reset.py`
    - `bulk_attendance_common.py` から `BULK_ATTENDANCE_API_WAIT_SECONDS` を `import` している
    - 出退勤時刻定数は参照していないけど、共通定数の命名変更をやるなら `import` 漏れの確認対象になる
- `tests/test_bulk_attendance_paid_holidays.py`
    - `19 * 60` などの期待値が固定就業時間前提になっている
- あたしの見立てだと、相棒が触りたい固定値は定数だけじゃなくて、判定ロジックとログ文言にも埋め込まれてるよ

### 2. 入力 UI の設計

- 入力順は以下の 3 つ
    - `対象月を入力してね ( yyyy-mm ):`
    - `出勤何時にする？ ( 0-23 の整数。Enter で 9 ):`
    - `退勤何時にする？ ( 0-23 の整数。Enter で 19 ):`
- 時刻入力は「時」だけ受ける。`9` と `09` は同じ扱いに正規化する
- `Enter` のみは `default` 扱いで、出勤は `09:00`、退勤は `19:00` を補完する
- 文字列バリデーションは `^\d{1,2}$` のみ許可する
- 数値バリデーションは `0 <= hour <= 23`
- 相関バリデーションは `start_hour < end_hour` を必須化する
- 失敗時は即時終了して `EXIT_CODE_MENU_ERROR` を返す
- エラーメッセージ案
    - `[エラー] 出勤時刻は 0-23 の整数で入力してね: 'xx'`
    - `[エラー] 退勤時刻は 0-23 の整数で入力してね: 'yy'`
    - `[エラー] 出勤時刻は退勤時刻より前にしてね ( start=09, end=09 )`
- 3 問目まで通ったら、内部では `work_start_minutes` と `work_end_minutes` に変換して後続へ渡す

### 3. 変更が必要なファイル / 関数の一覧

- `app/actions/bulk_attendance.py`
    - `handler`
        - 月入力の後に就業時刻入力を受け取り、日次処理へ引き回す
    - 新規ヘルパー案 `_parse_work_hours`
        - 出勤 / 退勤入力、 `default` 展開、バリデーション、 minutes 変換を担当する
    - `_process_date`
        - 引数に就業時刻設定を追加し、 payload 構築時の固定定数依存をなくす
    - `_build_default_work_record_payload`
        - 引数駆動に変更して `clock_in_at` / `clock_out_at` を可変化する
    - `_build_half_decision`
        - 半休判定境界を「入力された就業帯」で比較できる形に変更する
    - `_work_result_label`
        - ログ表示の勤務帯を入力値ベースに変更する
- `app/actions/bulk_attendance_common.py`
    - 今回は定数名をリネームしない
    - `BULK_ATTENDANCE_WORK_START_MINUTES` と `BULK_ATTENDANCE_WORK_END_MINUTES` を「デフォルト値」として継続利用する
- `tests/test_bulk_attendance_paid_holidays.py`
    - 既存テストを就業時刻引数ありの形へ更新する
    - カスタム時刻ケース ( 例: `08-17` ) を追加して回帰を抑える
- `app/actions/bulk_attendance_reset.py`
    - 実装変更はなし
    - ただし影響確認対象として扱い、共通定数の `import` が壊れていないことを確認する
    - 出退勤時刻の入力追加はしない ( リセット機能は削除専用のため )

### 4. 定数リネーム方針

- 今回スコープでは定数リネームを見送る
- 見送り理由は 3 つ
    - 今回の本質は「入力を受けて引数で流すこと」で、定数名変更は機能成立の必須条件じゃない
    - リネームを入れると `bulk_attendance.py` とテストの追従 diff が増えて、レビュー軸がぶれる
    - `bulk_attendance_reset.py` みたいに共通定数を `import` する別アクションの確認ポイントが増える
- 方針として、現行定数をデフォルト値として使い続ける
- 命名整理は別タスクのリファクタとして切り出す

### 5. 休憩時間への影響

- 結論として、調整は必要
- 理由は、固定 `11:30-12:30` を毎回送ると勤務帯外の休憩になって API 側で拒否されるリスクがあるため
- 当面は休憩基準時刻を `11:30-12:30` のまま維持する
- ただし送信条件を追加する
    - `work_start <= 11:30` かつ `work_end >= 12:30` の日だけ `break_records` を付与する
    - 条件を満たさない日は `break_records = []` で送信する
- これなら実装コストを増やしすぎずに、相棒の出退勤時刻カスタマイズを先に成立できる。現実解でいくよｗ
- 補足として、半休 `payload` は現状 `break_records` を送っていないため今回の変更影響はない

### 6. エラーハンドリング

- 入力フェーズ
    - 空入力は `default` 展開するのでエラーにしない
    - 非数値 / 範囲外は即時エラー終了
    - `出勤 >= 退勤` は即時エラー終了
- 日次処理フェーズ
    - API エラーは既存の `_print_api_error` 形式で日付とステップを明示して出力する
    - 休憩省略が起きた日は `[INFO]` ログを出して、想定内の挙動か追跡しやすくする
    - API エラー発生時は現行どおりその日で中断し、終了コード `1` を返す
- 想定外フェーズ
    - minutes 変換後に `start >= end` が発生したら `ActionExecutionError` を投げて防御する
- あたし的にはここまで押さえれば、相棒が時間帯をいじっても事故りにくい設計になる♡
