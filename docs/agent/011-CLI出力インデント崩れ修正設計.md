# 011 CLI出力インデント崩れ修正設計

## このノートの位置づけ

- `001-このプロジェクトの概要書` の責務分離方針に従い、`menu` 層の表示境界で問題を解消する
- 今回は設計のみを更新し、実装は行わない
- 操作仕様 (`↑` / `↓` / `Enter` / `Ctrl+C`) は維持する

## 要件

- メニュー選択後に出る `access_token:` が右にずれる問題を解消する
- メニュー選択後に出る `対象月を入力してね (yyyy-mm):` が右にずれる問題を解消する
- 以後の CLI 出力は左端揃え（不要インデントなし）を保証する

## 現象の再現条件

前提:
- `token.json` が存在し、メニュー表示まで到達できること

再現手順:
1. `docker compose run --rm app pipenv run python main.py` を実行する
2. メニューで `あ、いや、アクセストークン取得までいけるか見たかっただけ` を選ぶ（`Enter`）
3. `access_token: ...` の先頭位置が左端ではなく右へずれて出ることがある
4. もう一度 `docker compose run --rm app pipenv run python main.py` を実行する
5. メニューで `指定の月の平日に勤怠を詰め込む` を選ぶ（`Enter`）
6. `対象月を入力してね (yyyy-mm):` の先頭位置が右へずれて出ることがある

## 原因仮説（menu 描画後のカーソル位置）

- `app/menu/controller.py` の `select_menu_action()` は、`key == "enter"` で `return items[selected_index]` しており、確定直前/直後の後処理を持たない
- `app/menu/renderer.py` の `render_menu(..., initial=False)` は `sys.stdout.write("\r")` と `sys.stdout.write(f"\x1b[{len(items)}F")` で再描画位置へ戻すが、これは「次回メニュー再描画」のための復帰であり「メニュー終了時の復帰」ではない
- 同ファイルの `_write_menu_line()` は `sys.stdout.write(f"\r{prefix} {label}\x1b[K\n")` で 1 行ずつ描画するが、メニュー終了時に left edge を確定させる API が存在しない
- そのため、端末や PTY 経路によってはメニュー描画直後のカーソル列が `column 1` に戻りきらず、次の `print()` / `input()` がその列から始まり右ずれして見える
- 特に `docker compose run --rm app` の対話 TTY では、この差分が見えやすい

## 修正方針（どの層で left edge に戻すか）

方針:
- `menu` 層の終了境界で left edge へ戻す
- `actions` 層（`print_access_token`, `bulk_attendance` など）にはカーソル制御責務を持たせない

具体案:
1. `renderer` に「メニュー終了時のカーソル正規化」ヘルパーを追加する  
   例: `normalize_after_menu_exit()` で `\r\x1b[K` を必ず出力して `column 1` を保証する
2. `controller.select_menu_action()` は `raw_stdin_mode()` 内の `try/finally` で終了フックを持ち、`Enter` return 経路と `Ctrl+C` 例外経路の両方で `normalize_after_menu_exit()` を必ず実行する
3. `\x1b[K` は任意ではなく必須とする（left edge へ戻すだけでなく、行末側の残り表示を確実に消すため）

責務境界（確定）:
- `renderer` の責務: エスケープシーケンスの組み立てと実際の描画/正規化出力（`render_menu`, `_write_menu_line`, `normalize_after_menu_exit`）
- `controller` の責務: 入力イベントに応じた描画タイミング制御とライフサイクル管理（いつ `renderer` の正規化を呼ぶかの保証）
- `actions` / `bootstrap` の責務: 業務処理・オーケストレーションのみ（カーソル制御を持たない）

非採用方針:
- `actions` 側で都度 `print("\r...")` する
- `bootstrap` 側で action 実行前に一律補正する

非採用理由:
- 問題の起点は `menu` 描画境界であり、`actions` 補正は機能追加ごとに重複実装・漏れを生みやすい
- `bootstrap` 一律補正はメニュー非経由フロー（認証エラー出力など）にも副作用を広げ、責務境界を曖昧にする
- `001` の方針（表示は `menu`、業務は `actions`）に反するため、`menu` 層で完結させる設計を採用する

## 変更対象ファイル案

1. `app/menu/controller.py`
   - `Enter` 確定時の return 前に left edge 正規化を呼ぶ
2. `app/menu/renderer.py`
   - メニュー終了時のカーソル正規化ヘルパーを追加する
3. `docs/agent/011-CLI出力インデント崩れ修正設計.md`
   - 本設計内容を更新する（今回実施）

## 既存挙動（↑↓/Enter/Ctrl+C）への影響有無

- `↑` / `↓`: 影響なし（選択インデックス移動ロジックは不変）
- `Enter`: 挙動変更なし（選択確定は同じ）。追加は「確定直前の left edge 正規化」のみ
- `Ctrl+C`: 意味は不変（中断）。終了コード `130` の設計は維持
- OAuth / token 更新 / API 呼び出し: 影響なし

## 検証手順（実装後）

1. `docker compose up -d --build`（初回または環境更新時）
2. `docker compose run --rm app pipenv run python main.py`
3. メニューで `あ、いや、アクセストークン取得までいけるか見たかっただけ` を選択し、`access_token:` が左端開始であることを確認
4. 再度 `docker compose run --rm app pipenv run python main.py`
5. メニューで `指定の月の平日に勤怠を詰め込む` を選択し、`対象月を入力してね (yyyy-mm):` が左端開始であることを確認
6. `↑` / `↓` を連続操作（20回以上）した後でも、`Enter` 後の次出力が左端開始であることを確認
   - 根拠: 現在のメニューは 3 項目で、1キー入力ごとに全項目再描画されるため、20回で少なくとも 60 行以上の再描画が発生し、累積ドリフトの検出に十分な負荷となる
7. `Ctrl+C` 中断を確認し、終了コードが `130` であることを確認
   - 例: `docker compose run --rm app pipenv run python main.py; echo $?`
8. 非 TTY 条件を確認し、既存どおりメニューエラー（終了コード `2`）になることを確認
   - 例: `docker compose run --rm -T app pipenv run python main.py; echo $?`

## レビュー

### 1st review

#### 良い点

- 問題の再現条件が手順付きで明確
- 責務分離の方針 (`menu` 層で完結) が 001 概要書と一貫している
- Enter / ↑↓ / Ctrl+C の既存挙動への影響を明記
- 検証手順が具体的で、非 TTY 条件も含んでいる

#### 指摘事項

1. 原因仮説がやや弱い（renderer 実装箇所の具体参照が不足）
2. 修正方針の「必要なら `\x1b[K`」が曖昧（必須/任意を確定した方がよい）
3. Ctrl+C 経路の正規化が「実装時最終判断」で止まっている（設計段階で確定推奨）
4. renderer と controller の責務境界をもう一段明記した方がよい
5. 検証手順の連続入力回数の根拠を補足した方がよい
6. 非採用方針（actions/bootstrap 側補正）の非採用理由を補強した方がよい

### 対応ログ

1. 原因仮説へ `renderer/controller` の該当処理（`return items[selected_index]`, `\x1b[{len(items)}F`, `\r... \x1b[K\n`）を具体追記した。  
2. `\x1b[K` を任意ではなく「必須」に確定し、採用理由（行末残骸の除去）を明記した。  
3. `Ctrl+C` 経路は「実装時判断」を廃止し、`controller` の `try/finally` で常時正規化実行に確定した。  
4. `renderer` と `controller` の責務境界を設計本文へ明文化した。  
5. 連続入力 20 回の根拠（3項目 x 全再描画で 60 行以上）を検証手順に追記した。  
6. `actions/bootstrap` 非採用理由を、重複実装・副作用・責務逸脱の観点で補強した。  

#### 判定

判定: LGTM 依頼

#### 全体評価

- 1st review 指摘（原因具体化、`\x1b[K` 必須化、Ctrl+C 経路確定、責務境界明文化、検証根拠、非採用理由）が設計本文に反映済み。
- `001` の責務分離方針に沿って、`menu` 層で完結する修正方針になっている。
- 再現条件・検証手順・影響範囲が具体的で、実装者が判断しやすい。

#### 細部確認

- `renderer` 側の正規化ヘルパー追加と、`controller` 側 `try/finally` での常時呼び出しは妥当。
- `\x1b[K` 必須化は行末残骸対策として適切。
- `Ctrl+C` でも正規化を実行しつつ終了コード `130` を維持する設計は一貫している。
- 非採用方針（actions 補正 / bootstrap 一律補正）の理由が明確。

#### 判定

判定: LGTM
