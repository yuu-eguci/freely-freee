# 019 Terminal出力のログファイル保存設計

## 要望

- このツールを同僚にも使ってもらえるようにしたい。
- エラー時に「このフォルダのログファイルを送って」と案内できるようにしたい。
- そのため、実行時に Terminal に出る内容を、同じ内容のログファイルにも保存したい。
- いつ実行したログか分かるように、実行時刻を残したい。

## 設計方針（既存動作を壊さない）

- 既存の `print()` 挙動は変えず、`main.py` の入口で `stdout/stderr` を「Terminal + ファイル」の二重出力に差し替える。
- 既存の終了コード運用（`0 / 1 / 2 / 130`）はそのまま維持する。
- 既存の `app/bootstrap.py` 側ロジックには手を入れず、ログ機能は薄い共通部品として追加する。

## 詳細設計

### 1. ログ出力先ディレクトリ/ファイル命名規則

- ルート: `./logs/`
- 日付ディレクトリ: `./logs/YYYYMMDD/`
- ログファイル名: `freee-cli_YYYYMMDD_HHMMSS_pid<プロセスID>.log`
- 例: `logs/20260330/freee-cli_20260330_153045_pid12345.log`
- 時刻はローカル時刻（実行環境のタイムゾーン）で作る。
- 実装時は起動直後に `mkdir(parents=True, exist_ok=True)` で作成する。
- ファイルオープンは `encoding="utf-8"` を明示する（日本語ログの文字化け防止）。

### 2. Terminal + ファイルへ同時出力する方式

- `app/logging/tee_logger.py`（新規想定）に `TeeStream` を用意する。
- `TeeStream` は `write()` / `flush()` / `isatty()` を持ち、書き込み先を「元の Terminal ストリーム」と「ログファイル」の2つに流す。
- `main.py` で実行開始時に:
1. ログファイルを開く
2. 元の `sys.stdout` / `sys.stderr` を退避
3. `sys.stdout = TeeStream(original_stdout, log_file)`
4. `sys.stderr = TeeStream(original_stderr, log_file)`
- これで既存の `print()` 呼び出しは変更なしで二重出力される。
- `isatty()` は常に `original_stream.isatty()` を返す（TTY 判定の既存互換を維持）。
- `write()` の失敗方針:
  - Terminal 側への書き込みは最優先で継続する。
  - ログファイル書き込みに失敗したら、その時点でファイル出力のみ無効化する。
  - 失敗通知は一度だけ `original_stderr` へ `[LOG WARNING]` として出す。
  - ログ書き込み失敗を理由にアプリ全体は落とさない（既存動作優先）。
- 実行終了時は `finally` で `flush` とクローズを行い、`sys.stdout/stderr` を元に戻す。

### 3. stdout/stderr と例外時の扱い

- `stdout` と `stderr` は分離したまま扱う（色分けや既存運用に影響を出さない）。
- どちらも同一ログファイルへ保存する（時系列で追えるようにする）。
- ANSI エスケープコードは加工せずそのまま保存する（要望「Terminal出力をそのままログにも保存」を優先）。
- ハンドリング済み例外（既存の `run()` が終了コードを返すケース）:
  - 既存どおり `bootstrap.py` が `stderr` に出した内容をそのままログに残す。
- 未ハンドル例外:
  - `main.py` 側で `traceback.print_exc(file=sys.stderr)` を出す。
  - その後 `EXIT_CODE_APP_ERROR (1)` で終了する。
- `KeyboardInterrupt` は既存方針優先で `130` を維持し、中断メッセージもログに残す。
- シグナル時の保証範囲:
  - `KeyboardInterrupt`（通常の Ctrl+C）では `finally` に入るため、終了行とクローズは実行される。
  - `SIGTERM` などの外部シグナルは専用ハンドラ未実装のため、終了行・クローズの実行は保証対象外とする。
  - `SIGKILL` はプロセス即時終了のため、終了行やクローズの保証対象外とする。

### 4. 実行開始・終了時刻、終了コードの記録方針

- ログ冒頭に開始行を必ず出力:
  - 例: `[RUN START] 2026-03-30 15:30:45 +0900`
- ログ末尾に終了行を必ず出力:
  - 例: `[RUN END] 2026-03-30 15:31:12 +0900`
  - 例: `[EXIT CODE] 1`
- 実装は `main.py` で以下の順:
1. `start_at = now`
2. `run()` 実行して `exit_code` 受け取り
3. 未ハンドル例外時は `traceback.print_exc()` 出力後に `exit_code = 1` を確定
4. `finally` で `end_at` と `[EXIT CODE]` を出力
- 例外で `run()` が値を返せなかった場合も、`exit_code=1` を補完して記録する。
- `finally` 内でファイル出力できない場合でも、最低限 `original_stderr` に終了情報を出して終了する。

### 5. 同僚に渡すための運用メッセージ

- 実行の最後（`finally`）に、Terminal とログ両方へ次を出す:
  - `ログ保存先: <絶対パス>`
  - `不具合調査を頼むときは、この .log ファイルをそのまま送ってね`
- `Ctrl+C` 中断時も同じタイミングで上記メッセージを出す（ログファイル作成済みの場合）。
- ログファイル作成前に失敗した場合のみ、`ログ保存先` ではなく `ログファイルの作成に失敗したよ` を Terminal に出す。
- 同僚への案内は「フォルダ」より「ファイル単位」を基本にする（送付ミス防止）。
- もし複数回実行した場合は「対象時刻のファイル名（YYYYMMDD_HHMMSS）」を指定してもらう運用にする。

### 6. 終了コードごとの代表発生条件

- `0`: 正常終了（メニュー処理やアクションが完了）
- `1`: アプリケーションエラー（設定エラー、認証失敗、APIエラー、未ハンドル例外）
- `2`: メニュー系エラー（入力値不正、非対話環境など `MenuInputError` / `MenuEnvironmentError`）
- `130`: ユーザー中断（`MenuCancelled`、`KeyboardInterrupt`）

### 7. ローテーション/保持方針（最小）

- 保持期間は 30 日を基準にする。
- 起動時に `logs/YYYYMMDD/` 形式ディレクトリを走査し、30 日より古い日付ディレクトリは削除対象とする。
- 削除は best effort（失敗しても警告のみで続行）にする。
- ローテーション/削除処理が失敗しても本体処理の終了コードは変えない。

### 8. 配置先の根拠と変更対象（実装フェーズ想定）

- `./logs/` をリポジトリ直下に置く根拠:
  - 同僚に「このログファイルを送って」と案内しやすい。
  - `app/` 配下（実装コード）と運用生成物（ログ）を分離できる。
  - 既存の `token.json` と同じく、実行成果物をプロジェクト直下で扱う運用に揃えやすい。
- `app/logging/tee_logger.py` を新設する根拠:
  - `main.py` に実装を直書きせず、責務を分離できる。
  - 既存 `app/*` 構成（機能ごとのモジュール分離）と同じ方向性で管理できる。
- 変更対象:
  - `main.py`（起動時のログ初期化、開始/終了行、終了コード記録、運用メッセージ）
  - `app/logging/tee_logger.py`（新規: 二重出力ストリーム）
  - `tests/` 配下のログ機能テスト（新規）

### 9. テスト観点

- ログファイル作成:
  - 実行ごとに `logs/YYYYMMDD/` に1ファイル作成される。
  - ファイル名に `YYYYMMDD_HHMMSS` と `pid` が含まれる。
  - ログファイルが `utf-8` で読み出せる。
- 同時出力:
  - `stdout` 出力が Terminal とファイルで同一内容になる。
  - `stderr` 出力も Terminal とファイルで同一内容になる。
  - `isatty()` が元の Terminal 判定を維持する。
- ANSI:
  - ANSI エスケープを含む出力が加工されずログに残る。
- ログ書き込み失敗:
  - ディスクフル等を模擬したとき、`[LOG WARNING]` を1回だけ出して処理継続する。
- 例外/終了コード:
  - 正常終了で `[EXIT CODE] 0` が出る。
  - `MenuInputError` など既存エラー系で既存終了コード（`1/2/130`）が維持される。
  - 未ハンドル例外時に `traceback` と `[EXIT CODE] 1` が残る。
- 時刻記録:
  - `[RUN START]` と `[RUN END]` が両方ある。
  - `RUN END >= RUN START` になっている。
- 運用メッセージ:
  - 正常終了時、エラー終了時、`Ctrl+C` 時の全ケースで `ログ保存先` メッセージが出る（作成失敗時の代替メッセージ含む）。
- 保持方針:
  - 30日より古いログディレクトリ削除が best effort で動作する。
- 既存回帰:
  - 既存機能の標準出力メッセージ（例: トークン更新メッセージ、メニュー表示）が改変されていない。

## オーナーメモ

- 使い方が簡単で、トラブル時にログ回収しやすい形が理想。
- 既存の表示体験（Terminal 出力）は壊さない。

## レビュー

### 概要
設計として非常によく整理されており、要求仕様の理解から実装方針・テスト観点まで網羅的に記載されている。Terminal 出力とログファイル保存の二重出力という実装戦略は妥当で、既存コードへの影響を最小化する方針が明確。

### 指摘事項

- ログファイルのクローズ処理について
    - 「3. stdout/stderr と例外時の扱い」と「4. 実行開始・終了時刻、終了コードの記録方針」に `finally` でのクローズとフラッシュが記載されているが、`KeyboardInterrupt` (Ctrl+C) 時やシグナル受信時にも必ずクローズできるかを補足すべき。特に `SIGKILL` 以外のシグナル処理で正常終了行とファイルクローズが確実に実行される保証の有無を明記すること。
    - 対応内容: 採用。詳細設計 3 と 4 に保証範囲を追記した。`Ctrl+C` は `finally` 実行対象、`SIGTERM/SIGKILL` は保証対象外として明示した。
- `TeeStream.isatty()` の実装意図が不明確
    - Terminal 判定メソッドを持たせる理由とその戻り値の挙動 (元のストリームの `isatty()` を返すのか、常に `False` を返すのか) を明記すべき。既存コードが `sys.stdout.isatty()` に依存している場合、挙動変更で副作用が生じる可能性がある。
    - 対応内容: 採用。詳細設計 2 に `isatty()` は `original_stream.isatty()` を返す方針を明記した（既存互換優先）。
- ログファイルのエンコーディング指定なし
    - 日本語メッセージを含む出力がログファイルに正しく保存されることを保証するため、ログファイルオープン時のエンコーディング (例: `utf-8`) を明記すべき。
    - 対応内容: 採用。詳細設計 1 に `encoding="utf-8"` 明示を追記した。
- 未ハンドル例外時の終了コード記録タイミング
    - 「4. 実行開始・終了時刻、終了コードの記録方針」で「例外で `run()` が値を返せなかった場合も、`exit_code=1` を補完して記録する」とあるが、`traceback.print_exc()` を実行した後に `[EXIT CODE] 1` を出力する順序を明確にすべき。また、`finally` 内で例外が再発生した場合のフォールバック処理 (ログファイルが既にクローズされている場合など) の有無を検討すべき。
    - 対応内容: 採用。詳細設計 4 に「`traceback` 出力 -> `exit_code=1` 確定 -> `finally` で `[EXIT CODE]` 出力」の順序を追記し、`finally` 失敗時は `original_stderr` へフォールバックする方針を明記した。
- ログファイルのローテーションポリシーなし
    - `logs/YYYYMMDD/` 配下に日次でファイルが蓄積されるが、古いログの削除ポリシー (例: 30 日以上経過したログを削除) やディスク容量の考慮が一切ない。同僚への展開を考えるなら、運用開始後のディスク枯渇リスクと対処方針を記載すべき。
    - 対応内容: 採用。詳細設計 7 を新設し、30日保持・古い日付ディレクトリ削除・best effort 続行を明記した。
- `TeeStream.write()` のバッファリング動作
    - Terminal とファイルの両方に `write()` する際、片方が失敗した場合の挙動 (例: ディスクフルでファイル書き込み失敗) が不明確。Terminal 出力は継続するのか、例外を上げてプログラム全体を止めるのかを明記すべき。
    - 対応内容: 採用。詳細設計 2 に「Terminal 出力を優先継続、ログ書き込みのみ無効化、警告1回、アプリは継続」を追記した。
- ログ保存先メッセージの出力タイミング
    - 「5. 同僚に渡すための運用メッセージ」で「実行の最後に」とあるが、`KeyboardInterrupt` などの中断時にもこのメッセージを出すのかを明確にすべき。また、`finally` で出力するなら、その旨を明記すること。
    - 対応内容: 採用。詳細設計 5 に `finally` 出力で統一すること、`Ctrl+C` 時も出すこと、作成失敗時の代替メッセージを追記した。
- テスト観点の「同一内容」の定義が曖昧
    - 「7. テスト観点」の「`stdout` 出力が Terminal とファイルで同一内容になる」とあるが、ANSI カラーコードや制御文字の扱いが不明確。ログファイルにも色コードを残すのか、除去するのかを明記すべき (色コードを残すと可読性が下がる可能性)。
    - 対応内容: 採用。詳細設計 3 と 9 に「ANSI は加工せずそのまま保存」を明記し、テスト観点へ追加した。
- 既存エラーハンドリングの終了コード `2` の具体例なし
    - 「3. stdout/stderr と例外時の扱い」で `0 / 1 / 2 / 130` に言及しているが、`EXIT_CODE_MENU_ERROR (2)` が具体的にどのケースで返されるかの例がない。設計書として、各終了コードの発生条件を一覧化すると理解しやすい。
    - 対応内容: 採用。詳細設計 6 を新設し、`0/1/2/130` の代表発生条件を整理した。
- `app/logging/tee_logger.py` の配置根拠なし
    - 新規ファイル `app/logging/tee_logger.py` を作る方針だが、既存コードベースに `app/logging/` ディレクトリが存在しない場合、この配置が適切かどうかの根拠がない。既存のディレクトリ構成との整合性を確認すべき。
    - 対応内容: 採用。詳細設計 8 に配置理由を追記した。`main.py` への直書きを避けて責務分離し、既存の `app/*` 分割方針と整合を取る。

### 追加推奨事項

- パフォーマンスへの影響
    - 大量の `print()` が発生する処理 (例: API レスポンスの詳細ログ) で、ファイル I/O が性能ボトルネックにならないかを検証する必要がある。特に `flush()` の頻度によってはディスク I/O 待ちが発生する可能性がある。
    - 対応内容: 非採用（今回スコープ外）。今回は既存挙動維持を優先し、性能最適化は別タスクで扱う。なお、詳細設計 9 にログ機能回帰テストを追加して最低限の品質担保は入れた。
- 既存コードの `sys.stdout` 直接参照への影響
    - `main.py` で `sys.stdout` を差し替えるため、`bootstrap.py` や action 内で `sys.stdout` を直接参照している箇所があれば、その挙動が変わる可能性がある。事前に全コードを検索して影響範囲を確認すべき。
    - 対応内容: 採用。詳細設計 2 の `isatty` 互換方針と、詳細設計 9 の既存回帰観点で影響確認する前提を明記した。
