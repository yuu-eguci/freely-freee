# このプロジェクトの概要書

このドキュメントは、最初にこのプロジェクトへ入る人向けの最短ガイドです。  
実装判断は `docs/agent/004-mainpyリファクタリングとメニュー拡張設計.md` を正として進めます。

## プロジェクトの目的

- freee 連携の CLI を安全に拡張し続ける
- メニュー項目追加時に `main.py` を太らせない
- 認証・表示・業務処理・API 通信の責務混在を防ぐ

## 現在の実装方針（確定）

- `main.py` は薄い入口にする
- 実行フローは `app/bootstrap.py` の `run() -> int` に集約する
- `app` 配下で責務分離する
- メニュー (`menu`)・業務アクション (`actions`)・API 通信 (`clients`) を分離する
- メニュー拡張は `registry` 登録方式で行う
- 大規模置換はせず、段階移行で安全に進める
- 終了コードは `0 / 1 / 2 / 130` を維持する

## 責務の見取り図

```text
main.py
  -> app/bootstrap.py
     -> app/auth/*        # OAuth / token 管理
     -> app/menu/*        # メニュー表示と入力
     -> app/actions/*     # 選択機能ごとのユースケース
     -> app/clients/*     # freee API 通信共通化
```

## 実装ルール（必読）

- `main.py` に業務処理を書かない
- `menu` 層で API 通信しない
- `actions` は `AppContext` を受け取り、1 機能 1 モジュールを基本にする
- freee API 呼び出しは `clients` 経由に寄せる
- 新機能追加時に `if/elif` の巨大分岐を再導入しない
- `refresh_token` の責務は `auth` 層に閉じる

## メニュー機能を 1 つ追加する手順

1. `app/actions/<action_name>.py` を追加する
2. `handler(context: AppContext) -> int` を実装する
3. `app/actions/registry.py` に `ActionDefinition` を登録する
4. 必要に応じて `app/clients/` に API 呼び出しを追加する

## 段階移行の考え方

- 既存機能を壊さないことを最優先に、hybrid で段階的に移行する
- 先に共通定義と境界を作り、その後に機能を 1 つずつ移す
- 「動く状態を保ちながら小さく進める」を守る

## 終了コード

- `0`: 正常終了
- `1`: アプリケーションエラー（設定/認証/API/実行失敗）
- `2`: 対話入力・端末条件のエラー
- `130`: ユーザー中断（Ctrl+C など）
