# 005 指定日付の勤怠記録とタグ付与設計

## 要望

- 今回は実装せず、設計のみを作成する
- `yyyy-mm-dd` 形式で指定した日付に、`09:00` から `19:00` の勤怠を記録したい
- freee の「勤怠タグ」「日時タグ」に「出社」タグを記録したい
- API レスポンス構造が不明なので、必要なら事前に ID 取得を行う複数段階の実行フローでもよい
- API ドキュメント参照元は `002` ノート

## この設計のゴール

指定日付 1 日分について、freee 人事労務 API を使って以下を安全に実行できる設計を作る。

- 対象従業員の当日勤怠を `09:00-19:00` で登録または更新する
- 同じ日について「勤怠タグ」「日時タグ」に `出社` を設定する
- API 構造や必須 ID が不明でも、事前探索で必要情報を確定してから本実行へ進める

今回は設計だけ。CLI 実装、HTTP クライアント実装、`.env` 読み書き実装は次回以降に回す。

## 参考にする前提

- 認証、アクセストークン取得、`company_id` の基本取得は `002` ノートの方針を使う
- プロジェクト全体の実装責務は `001` と `004` の方針に従う
- freee Developers Community の人事労務 API 公開情報では、勤怠取得/更新系として少なくとも以下のエンドポイント名が確認できる
  - `GET /api/v1/employees/{employee_id}/work_records/{date}`
  - `PUT /api/v1/employees/{employee_id}/work_records/{date}`
- ただし、レスポンス JSON の詳細構造、タグ項目名、打刻区分項目名、タグマスタ取得 API の有無はこのノート時点では未確定

## まず押さえる前提整理

今回やりたいことは、実は 1 回の API 呼び出しで完了するとは限らない。
理由は 3 つある。

1. `work_records/{date}` の更新 payload に、`出社` タグを直接含められるか未確定
2. `出社` を表す値が、タグ ID なのか、タグ名なのか、コード値なのか未確定
3. `09:00` と `19:00` が、単純な開始終了時刻なのか、打刻区分付きの配列なのか未確定

そのため今回は、以下の三段階方式で設計する。

1. 事前取得ステップ
2. `.env` 反映ステップ
3. 本実行ステップ

この分け方なら、API の実レスポンスを 1 回見れば以後の運用をかなり安定させやすい。

## 想定する必要 ID / コード一覧

### 確定で必要そうなもの

| 項目 | 用途 | 取得元の第一候補 |
|---|---|---|
| `company_id` | 対象事業所の特定 | `GET /api/1/users/me?companies=true` または `GET /api/1/companies` |
| `employee_id` | 勤怠対象の従業員特定 | 人事労務 API の従業員一覧/従業員詳細系 |
| `date` | 勤怠対象日 | CLI 入力値 |

### 高確率で必要そうなもの

| 項目 | 用途 | 仮説 |
|---|---|---|
| `attendance_tag_id` または同等値 | 勤怠タグ `出社` の指定 | 勤怠タグ一覧 API があればそこから取得。なければ既存 `work_record` から逆引き |
| `datetime_tag_id` または同等値 | 日時タグ `出社` の指定 | 日時タグ一覧 API があればそこから取得。なければ既存 `work_record` から逆引き |
| `clock_in_type_id` / `breakdown type` / `time_clocks[].type` 相当 | `09:00` が「出勤」であることの指定 | 既存勤怠レスポンスから型名を確認 |
| `clock_out_type_id` / `time_clocks[].type` 相当 | `19:00` が「退勤」であることの指定 | 既存勤怠レスポンスから型名を確認 |

### 場合によって必要なもの

| 項目 | 用途 | 仮説 |
|---|---|---|
| `work_record_id` | 更新時に内部レコード ID が必要な場合 | `GET work_records/{date}` から取得 |
| `employee_number` や `freee user id` | `employee_id` 探索補助 | 従業員一覧 API の検索キーとして使用 |
| `tag_group_id` | 同名タグ衝突の解消 | タグ一覧 API がグループ返却する場合に使用 |

## API / データ取得の探索順

レスポンス構造が不明な前提なので、最初から本番更新を打ちにいかず、読み取り API を使って順番に埋める。

### Step 0. 認証前提の確認

- `002` ノートの方式でアクセストークンを取得できること
- `.env` に最低限以下があること
  - `FREEE_CLIENT_ID`
  - `FREEE_CLIENT_SECRET`
  - `FREEE_REDIRECT_URI`
  - `FREEE_COMPANY_ID` または後で入れる前提
- `token.json` に有効な `refresh_token` があること

ここが崩れていたら、勤怠設計以前に認証再取得へ分岐する。

### Step 1. 会社 ID の確定

#### 使う API

- `GET https://api.freee.co.jp/api/1/users/me?companies=true`
- 予備: `GET https://api.freee.co.jp/api/1/companies`

#### 目的

- ログイン中ユーザがアクセスできる `company_id` を確定する
- `.env` の `FREEE_COMPANY_ID` が存在するなら、それが実在するか確認する

#### 確認ポイント

- `companies` 配列の有無
- 会社ごとの `id`, `display_name` 相当の項目名
- 事業所が複数ある場合の選択ルール

#### 設計判断

- `FREEE_COMPANY_ID` 未設定なら、初回探索モードで候補一覧を表示して `.env` に反映する
- `FREEE_COMPANY_ID` 設定済みでも API 上で見つからなければ不正設定扱いにする

### Step 2. 従業員 ID の確定

#### 使う API

- 第一候補: 人事労務 API の従業員一覧 API
- 第二候補: ログインユーザ情報と紐づく従業員参照 API
- 予備: 既存勤怠 API 呼び出しに必要な一覧 API を周辺ドキュメントから追加探索

#### 目的

- 勤怠を登録する対象の `employee_id` を確定する

#### 確認したい項目

- `employee_id`
- 社員番号
- 氏名
- メールアドレス
- 在籍状態
- 所属会社 ID

#### 設計判断

- `.env` に `FREEE_EMPLOYEE_ID` を保持する前提にする
- 初回は従業員一覧を取得して、以下の優先順位で 1 人に絞る
  1. `.env` に `FREEE_EMPLOYEE_ID` がある
  2. `.env` に氏名やメールアドレスなど補助キーがあり一意に一致する
  3. 一覧を表示して人間が確認し `.env` に反映する
- 同名従業員がいる前提で、氏名だけで自動決定しない

### Step 3. 既存勤怠レコードの構造確認

#### 使う API

- `GET /api/v1/employees/{employee_id}/work_records/{date}`

#### 目的

- 更新対象日付の JSON 構造を観察する
- どのフィールドに勤務時刻、勤怠タグ、日時タグ、打刻区分が入るかを確定する

#### この段階で見るべき仮説フィールド

- `work_record.id`
- `work_record.date`
- `time_clocks` またはそれに相当する配列
- `clock_in_at`, `clock_out_at` 相当の単独項目
- `attendance_tags`, `work_tags`, `tags` 相当の配列
- `datetime_tags`, `time_tags` 相当の配列
- 各タグの `id`, `name`, `code`
- 打刻区分の `type`, `type_id`, `kind`, `name`

#### 設計判断

- まずは「対象日付に既存データがある日」で検証するのが安全
- ここで返るレスポンスを元に、後続の env 項目名と更新 payload を確定する
- 対象日付にレコードがなくても、それ自体は異常ではない。`404` または空レコード返却のどちらかを観察対象にする

### Step 4. タグ ID の探索

#### 使う API の優先順位

1. 勤怠タグ一覧 API / 日時タグ一覧 API が公式に存在するなら、それを使う
2. 存在しない、または見つからない場合は `GET work_records/{date}` のレスポンスから既存タグを逆引きする
3. それでも見つからない場合は、人間が freee 管理画面でタグ設定を確認し、ID または名称体系を手動確定する

#### 目的

- `出社` に対応する値を、「勤怠タグ用」と「日時タグ用」で別々に確定する

#### 設計判断

- `.env` には以下のように分けて保持する前提にする
  - `FREEE_ATTENDANCE_TAG_NAME=出社`
  - `FREEE_DATETIME_TAG_NAME=出社`
  - `FREEE_ATTENDANCE_TAG_ID=`
  - `FREEE_DATETIME_TAG_ID=`
- まず名称を基準値として持ち、探索で ID が取れたら ID を優先利用する
- 「勤怠タグ」と「日時タグ」で同じ表示名 `出社` でも別 ID である可能性を前提にする

### Step 5. 打刻区分の探索

#### 使う API

- 第一候補: `GET work_records/{date}`
- 第二候補: 打刻区分マスタ取得 API があればそれ

#### 目的

- `09:00` を何の区分で送るか、`19:00` を何の区分で送るかを確定する

#### 想定パターン

1. `clock_in_at` / `clock_out_at` の 2 項目だけ送ればよい
2. `time_clocks` 配列に `[{type: "clock_in", datetime: ...}, {type: "clock_out", datetime: ...}]` のように送る
3. `type_id` を必要とするマスタ参照型

#### 設計判断

- 初回探索では既存の正常勤怠 1 日分を読み、そのフォーマットをそのまま真似る方針にする
- 仕様が曖昧なうちは、時刻フォーマットを想像で固定しない

## `.env` に持たせる想定項目

初回探索が終わったら、以後の運用を楽にするために `.env` に以下を保持する設計にする。

| 変数名 | 必須 | 説明 |
|---|---|---|
| `FREEE_COMPANY_ID` | 必須 | 対象事業所 ID |
| `FREEE_EMPLOYEE_ID` | 必須 | 勤怠対象従業員 ID |
| `FREEE_ATTENDANCE_TAG_NAME` | 必須 | 勤怠タグ表示名。初期値 `出社` |
| `FREEE_DATETIME_TAG_NAME` | 必須 | 日時タグ表示名。初期値 `出社` |
| `FREEE_ATTENDANCE_TAG_ID` | 任意 | 勤怠タグ ID。探索で確定したら設定 |
| `FREEE_DATETIME_TAG_ID` | 任意 | 日時タグ ID。探索で確定したら設定 |
| `FREEE_CLOCK_IN_TYPE` | 任意 | 出勤打刻区分。文字列型の場合に使用 |
| `FREEE_CLOCK_OUT_TYPE` | 任意 | 退勤打刻区分。文字列型の場合に使用 |
| `FREEE_CLOCK_IN_TYPE_ID` | 任意 | 出勤打刻区分 ID。ID 型の場合に使用 |
| `FREEE_CLOCK_OUT_TYPE_ID` | 任意 | 退勤打刻区分 ID。ID 型の場合に使用 |

ポイントは、タグや打刻区分を「名前しか分からない状態」から始めても運用できるようにしておくこと。
初回探索で ID が取れたら ID 利用に寄せる。

## 実行フロー設計

### フロー A. 事前取得モード

初回セットアップ、または ID が未確定なときだけ使う。

#### 入力

- 認証済みトークン
- 必要なら仮の日付 1 件
- 必要なら従業員候補を絞る補助情報

#### 実施順

1. `users/me` または `companies` から `company_id` を確定する
2. 従業員一覧 API から `employee_id` を確定する
3. `GET work_records/{date}` でレスポンス構造を観察する
4. タグのフィールド名、値型、ID 候補を抽出する
5. 打刻区分の型を抽出する
6. 確定した値を `.env` に反映する

#### 出力

- `.env` に確定値を保存
- 本実行に必要な未確定項目が残っていないか一覧化

### フロー B. `.env` 反映モード

これは API 実行というより構成確定フェーズ。

#### やること

- 事前取得で確定した ID / 名称 / 区分値を `.env` に保存する
- 次回以降は探索 API を毎回叩かず、本実行を短くする

#### 設計判断

- `.env` 反映は手動でもよいし、将来は補助コマンドで自動更新してもよい
- 今回は設計のみなので、書き換え主体は人間でも問題ない

### フロー C. 本実行モード

#### 入力

- `date` (`yyyy-mm-dd`)
- `09:00`
- `19:00`
- `.env` に入った `company_id`, `employee_id`, 各タグ値, 打刻区分値

#### 実施順

1. 入力日付を検証する
2. `employee_id` など必須 env を検証する
3. `GET work_records/{date}` を呼び、既存データ有無と構造を確認する
4. 既存構造に合わせて更新 payload を組み立てる
5. `PUT /api/v1/employees/{employee_id}/work_records/{date}` を実行する
6. レスポンスを再取得して、`09:00-19:00` と両タグ `出社` が入ったことを確認する

#### payload 組み立て方針

レスポンス構造が不明なため、以下の固定方式ではなく「観察して合わせる」方式にする。

- 時刻が単独項目なら、その項目を更新する
- 時刻が配列なら、既存配列の形式に従って `09:00` と `19:00` を設定する
- タグが ID 配列なら ID を送る
- タグがオブジェクト配列なら既存形式に従う
- タグが名前指定型なら `出社` の名称指定を使う

## 正常系の具体イメージ

### 期待する操作結果

- 指定日付の勤怠が存在しない場合
  - その日付の勤怠レコードが新規状態または空状態から `09:00-19:00` で埋まる
- 指定日付の勤怠が存在する場合
  - その日付の勤務開始・終了が `09:00`, `19:00` になる
  - `勤怠タグ` に `出社` が入る
  - `日時タグ` に `出社` が入る

### 日付と時刻の扱い

- CLI 入力は `yyyy-mm-dd` 固定
- 時刻は `09:00`, `19:00` 固定で設計する
- API が日時型を要求する場合は、対象日付と組み合わせてローカルタイムとして送る
- API がタイムゾーン付き文字列を要求するか、`HH:MM` のみでよいかは事前取得で確認する

## 失敗時の分岐設計

### 1. ID 未発見

#### パターン

- `company_id` が見つからない
- `employee_id` が見つからない
- `attendance_tag_id` / `datetime_tag_id` が見つからない
- 打刻区分が見つからない

#### 対応

- `company_id`, `employee_id` が未発見なら本実行に進まない
- タグ ID が未発見でも、名称指定が可能なら名称指定にフォールバックする
- タグ ID も名称指定も無理なら、freee 管理画面でタグ設定を人手確認して再実行する
- 打刻区分未発見なら、既存正常勤怠日を追加で観察して型を確定する

### 2. タグ名重複

#### パターン

- 勤怠タグに `出社` が複数ある
- 日時タグに `出社` が複数ある
- 勤怠タグと日時タグで見た目は同じだが別物が複数ある

#### 対応

- 自動で先頭一致を採用しない
- 候補一覧を出して、`id`, `name`, `group`, `enabled` など識別可能な属性で人間確認する
- 確定後は `.env` に ID を保存し、次回以降は名前解決を避ける

### 3. 日付フォーマット不正

#### パターン

- `2026-3-1` のようにゼロ埋めなし
- `2026/03/01` のように区切り違い
- 存在しない日付 (`2026-02-30`)

#### 対応

- 本実行前に `yyyy-mm-dd` を厳密検証する
- 不正なら API を呼ばずに終了する
- エラーメッセージでは、受け付ける形式を明示する
  - 例: `日付は 2026-03-26 のような yyyy-mm-dd 形式で入力してね`

### 4. API 権限不足

#### パターン

- `401 Unauthorized`
- `403 Forbidden`
- 会社閲覧はできるが勤怠更新だけ不可

#### 対応

- `401` はトークン期限切れを疑い、まず refresh を試す
- `403` はスコープ不足または freee 側権限不足として扱う
- 認証再取得で直るのか、freee アプリ設定の権限追加が必要なのかを切り分ける
- `users/me`, `companies`, `work_records GET`, `work_records PUT` のどこで失敗したかを出し分ける

### 5. 既存勤怠との衝突

#### パターン

- 既に別時刻が入っている
- 既に他タグが付いている
- API が部分更新ではなく全置換型

#### 対応

- いきなり盲目的に上書きしない
- `GET` 結果との差分を見て、必要な項目だけ変更する payload を組む
- API が全置換型なら、既存必要項目を保持したうえで目的項目を差し替える
- 未知項目は消さない方針を優先する

## 実装時の責務分離メモ

今回は実装しないけど、次回のために責務だけ置いておく。

- `auth`: トークン refresh と保存
- `clients`: `users/me`, `companies`, `employees`, `work_records` の API 呼び出し
- `actions`: 事前取得モード、本実行モードのユースケース
- `menu`: 日付入力や実行モード選択

`main.py` に勤怠ロジックを直書きしない点は `001` / `004` の方針どおり。

## この設計での結論

今回の要件は、最初から「指定日付に勤怠を 1 発登録する」設計にせず、以下の二段階運用にした方が安全。

1. 読み取り系 API で `company_id`, `employee_id`, タグ値, 打刻区分, `work_records` の JSON 構造を確定する
2. 確定した値を `.env` に反映したうえで、`PUT /api/v1/employees/{employee_id}/work_records/{date}` で `09:00-19:00` と `出社` タグ 2 種を更新する

この設計なら、API レスポンス構造が不明でも、探索結果を 1 回固定化すれば以後の本実行を安定させやすい。
また、タグ名重複や権限不足みたいな運用事故も、事前取得フェーズでかなり早く検知できる。

## レビュー観点

- `employee_id` の探索条件はこの設計で足りるか
- `出社` が「勤怠タグ」「日時タグ」の両方で本当に別管理か
- `PUT work_records/{date}` が部分更新か全置換かを、実レスポンス観察前提で進めて問題ないか
- `.env` に保持すべき識別子はこれで足りるか
- 初回探索で見るべきサンプル日付の取り方に追加条件が必要か
