# 016 機能名に「自分」を追加する変更設計

## 要望

- "指定の月に勤怠を詰め込む" -> "指定の月に自分の勤怠を詰め込む" に機能名変更
- "指定の月の勤怠をリセットする" -> "指定の月の自分の勤怠をリセットする" に機能名変更

## 詳細設計

- あたしの今回のゴールは、CLI メニューに出す機能名だけを「自分の勤怠」表記にそろえること。
- 相棒向けに先に結論を書くと、`app/` 配下で変更必須なのは `app/actions/registry.py` の 2 箇所。
- `013` は「指定月」表記を「指定の月」へそろえる変更で、今回はその続きとしてメニュー文言に「自分の」を加える差分。

### 1. 変更対象の洗い出し ( ソースコード )

- `app/actions/registry.py`
    - `action_id="bulk_attendance"` の `menu_label` を `指定の月に自分の勤怠を詰め込む` へ変更。
    - `action_id="bulk_attendance_reset"` の `menu_label` を `指定の月の自分の勤怠をリセットする` へ変更。
- それ以外の `app/` 配下ファイルは、今回の機能名文字列を直接保持していないため変更対象外。
- `app/actions/bulk_attendance.py` と `app/actions/bulk_attendance_reset.py` の docstring は今回は据え置き。
    - 判断理由: docstring は「処理内容の説明」で、メニュー表示名のユーザー向け文言とは責務が違うため。
    - `013` との関係: `013` の docstring 更新は「指定月 -> 指定の月」の旧表記是正が目的。今回は旧表記是正ではなく UI 文言の具体化なので、同一方針にはしない。

### 2. 変更対象の洗い出し ( 設計ノート )

- `docs/agent/016-機能名に自分を追加設計.md`
    - このファイルの `詳細設計` セクションに、変更箇所と docstring 判断とテスト方針を追記。
- `docs/agent/013-機能名変更設計.md`
    - 方針比較の参照元として扱う。履歴は固定し、今回タスクの更新対象にはしない。
- 過去設計ノート ( `014` `015` など ) は履歴として固定し、今回タスクの更新対象にしない。

### 3. テスト変更箇所の設計

- 今回は変更なし。
    - 当初案では `app.actions.registry.to_menu_items( debug_mode=False )` のラベル検証テスト追加を検討した。
    - ただし `1st review` で「静的文字列 2 箇所の置換に対しては過剰」と判断され、見送る方針に確定した。
    - `013` でも同スコープの名称変更でテスト追加なしだったため、運用方針ともそろう。

### 4. 影響範囲

- 影響範囲はメニュー表示テキストのみ。
- `action_id` とハンドラ実装は不変なので、実行フローと freee API 呼び出しロジックへの影響はない。
- docstring は今回据え置きなので、コード内説明文の差分は発生しない。

## レビュー

### 1st review

全体的にシンプルでよくまとまってる♡ ただ 2 点だけ指摘するね。

1. テスト追加は過剰だと思う
    - 今回は `registry.py` の文字列リテラルを 2 箇所書き換えるだけ。新テストファイル `tests/test_action_registry_menu_labels.py` を追加するほどじゃないｗ
    - `menu_label` は静的な文字列定数で、ロジック分岐がない。テストで守るべき複雑さがない。
    - 013 の機能名変更のときもテスト追加なしで通してるし、今回も同じスコープでいいと思う。
    - -> テストセクションは「今回は変更なし」にして、テスト追加案は削除してほしい。
    - 対応内容: `詳細設計 3` を「今回は変更なし」に更新し、当初は追加検討したが `1st review` で見送った経緯を明記した。

2. docstring も更新すべきか判断を明記してほしい
    - 設計で「docstring は対象を分ける」と書いてあるけど、もうちょっとはっきり書いてほしい。
    - `bulk_attendance_reset.py` の docstring `"指定の月の勤怠を一括でリセットする"` は、今回の変更後のメニュー名 `"指定の月の自分の勤怠をリセットする"` と微妙にずれる。
    - docstring を変えるか変えないか、明確に判断を書いてほしい。あたし的には「メニュー名のみ変更、docstring は据え置き」で OK だと思うけど、設計として明言しておくべき。
    - 対応内容: `詳細設計 1` に「docstring は据え置き」と判断を明記し、理由 ( docstring の責務は処理説明 ) を追記した。

上記 2 点を反映してくれたら LGTM 出せると思う♡

### 2nd review

1st review の指摘に全面的に賛成する♡ さらにあたしからも追加で指摘するね。

1. docstring の方針について、もう少し踏み込んだ意見
    - 1st review の「メニュー名のみ変更、docstring は据え置き」案に基本的には賛成だけど、013 との整合性が気になる。
    - 013 では `bulk_attendance.py` と `bulk_attendance_reset.py` の docstring も変更してるんだよね (「指定月」->「指定の月」)。
    - 今回も 013 と同じ方針を取るなら、docstring も「自分の勤怠」を追加すべき。
    - もし据え置きにするなら、「013 とは方針が異なる理由」を明記してほしい。
    - あたし的には 013 と揃えるほうが自然だと思うけどw
    - 対応内容: `詳細設計 1` に `013` との差分理由を追記した。`013` は旧表記是正、`016` は UI 文言具体化なので据え置きにする方針を明記した。

2. 過去設計ノートとの関係性を明記すべき
    - 016 の設計ノートに「過去設計ノートは履歴として固定し、今回タスクの更新対象にしない」って書いてあるのは良いんだけど、013 の存在に触れてないよね。
    - 今回の変更は 013 の続きみたいなものだから、「013 と同様の変更方針を取る」or「013 とは異なる方針を取る」を明記してほしい。
    - これがあると、レビュワーが過去の変更履歴と比較しやすくなるよ♡
    - 対応内容: `詳細設計` 冒頭と `詳細設計 2` に `013` との関係を追記し、参照はするが更新対象にはしないことを明記した。

3. テスト追加の判断について
    - 1st review の「テスト追加は過剰」という指摘に賛成。
    - ただ、設計セクション 3 で「テスト変更箇所の設計」としてかなり詳しく書いてあるから、削除するだけじゃなくて「当初はテスト追加を検討したが、1st review で過剰と判断されたため見送る」みたいな経緯を残しておくと、将来同じ議論をしなくて済むと思う。
    - 対応内容: `詳細設計 3` に「当初検討 -> 1st review で見送り確定」の経緯を追記した。

上記 3 点を反映してくれたら、あたしも LGTM 出せるよ♡
