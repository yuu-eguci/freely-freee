# 014 有給登録時の出社タグ除外設計

## 要望

"指定の月に勤怠を詰め込む" 機能の修正。

    - 有給として勤怠登録するときは、出社タグを登録しない
    - 半休のときは現状のまま ( 出社タグを登録する ) で OK

## 詳細設計

相棒、あたしが今回の変更ポイントを実装フローに沿って固めるねｗ

### 1. 現在の実装フローの確認 ( `app/actions/bulk_attendance.py` )

`handler()` は月入力後に `users/me` と `attendance_tags` を取得し、日付ループで `_process_date()` を呼ぶ。

`_process_date()` の主要フローはこう。

    - `get_work_record()` で対象日を取得し、`day_pattern == "normal_day"` と `use_default_work_pattern == True` の日だけ処理する。
    - `_decide_paid_holiday()` で `decision.kind` を決める ( `none` / `full` / `half` / `half_fallback` ) 。
    - `decision` に応じた `work_record` payload を `_build_work_record_payload()` で作り、`put_work_record()` を実行する。
    - その後に常に `put_attendance_tags()` を実行して「出社」タグを付与している。

今は `decision.kind` が `full` でも `put_attendance_tags()` が呼ばれる実装になってる。

### 2. 有給登録と判定する条件 ( どの変数 / フラグを見るか )

有給登録かどうかの判定には `_process_date()` 内の `decision.kind` を使います。

    - `decision.kind == "full"` を「全休有給として勤怠登録する日」と定義します。
    - `decision.kind == "half"` は半休なので今回の除外対象にしません。
    - `decision.kind == "half_fallback"` は通常勤務への fallback なので今回の除外対象にしません。
    - `decision.kind == "none"` は有給申請なしの通常勤務として扱うため、今回の除外対象にしません。

`decision.kind` 自体は `_decide_paid_holiday()` が `holiday_type` / `status` / `revoke_status` を見て決定しているため、今回のタグ除外条件は `decision.kind` だけで判定できます。

### 3. 出社タグをスキップするための変更箇所

変更箇所は `_process_date()` の `put_work_record()` 成功後です。現行コード ( `app/actions/bulk_attendance.py:273-283` ) は `tag_payload` を先に作って常に `put_attendance_tags()` を呼んでいるため、次の順序へ変更します。

    - `work_label = _work_result_label(decision)` を先に評価します。
    - `if decision.kind == "full":` を追加します。
    - `full` の場合は `put_attendance_tags()` を呼ばず、`print(f"[OK]   {date} {work_label} 勤怠登録済み (出社タグなし)")` を出して `return "success"` します。
    - `full` 以外のときだけ `tag_payload = _build_attendance_tag_payload(...)` を作成し、既存どおり `put_attendance_tags()` を実行します。
    - `put_attendance_tags()` の `except` 時ログは既存どおり `勤怠登録済み` を維持し、`_print_api_error()` の後に `return "error"` します。

この構成なら、勤怠登録 API ( `put_work_record` ) の成功を前提に、全休有給日だけタグ付与を確実にスキップできます。

### 4. タグ付与を継続するケースの明記

`full` 以外は現状維持でタグを付与します。

    - `decision.kind == "half"` の日は、これまでどおり `put_work_record()` 実行後に `put_attendance_tags()` を実行します。
    - `decision.kind == "half_fallback"` も現状どおりタグを付与します。
    - `decision.kind == "none"` は通常勤務 ( 09:00-18:00 ) として登録されるため、出社タグを付与します。
    - つまり「出社タグを外す」のは `full` の日だけ、という境界を固定します。

この境界でいけば、相棒の要望どおり「有給はタグなし、半休はタグあり」が成立します♡

#### 指摘1: 設計と実装のズレ ( タグ付与ロジックの順序 )

設計書 (セクション 3) には「`put_work_record()` 成功後から `put_attendance_tags()` 呼び出し前」に条件分岐を入れるって書いてあるけど、実際のコードは以下の流れになってる。

    274: work_label = _work_result_label(decision)
    275: try:
    276:     hr_client.put_attendance_tags(employee_id, date, tag_payload)
    277: except ApiResponseError as exc:
    278:     print(f"[OK]   {date} {work_label} 勤怠登録済み")
    279:     _print_api_error(date, "put_attendance_tags", exc)
    280:     return "error"
    281:
    282: print(f"[OK]   {date} {work_label} 出社タグ付与済み")
    283: return "success"

つまり、現状は常に `tag_payload` を作って `put_attendance_tags()` を呼んでる。

設計どおりに実装するなら、以下のように変更する必要がある。

    274:

    276:     print(f"[OK]   {date} {work_label} 勤怠登録済み")
    277:     return "success"
    278:
    279: tag_payload = _build_attendance_tag_payload(company_id, attendance_tag_id)
    280: try:
    281:     hr_client.put_attendance_tags(employee_id, date, tag_payload)
    282: except ApiResponseError as exc:
    283:     print(f"[OK]   {date} {work_label} 勤怠登録済み")
    284:     _print_api_error(date, "put_attendance_tags", exc)
    285:     return "error"
    286:
    287: print(f"[OK]   {date} {work_label} 出社タグ付与済み")
    288: return "success"

これなら `decision.kind == "full"` のときだけ、タグ付与を完全にスキップできる♡

対応内容:

#### 指摘2: ログメッセージの整合性

設計書では「`[OK]` ログを出して `success` で return する」と書いてあるけど、上記の実装例では `decision.kind == "full"` のときのログメッセージが「勤怠登録済み」だけになる。

有給のときは「出社タグを付けない」という意図が明確に分かるログにしたほうがユーザに親切だよ。

例えばこんな感じ。

    print(f"[OK]   {date} {work_label} 勤怠登録済み (出社タグなし)")

こうすれば、相棒が実行ログを見たときに「ああ、有給だからタグ付けてないんだな」ってすぐ分かるｗ

対応内容:

    - 妥当と判断して、詳細設計の セクション 3 に `full` 時のログ文言を明記しました。
    - 文言は `勤怠登録済み (出社タグなし)` を採用し、タグ未付与の意図が実行ログで判別できるようにしました。

#### 指摘3: `decision.kind == "none"` のケースの扱い

設計書の セクション 2 と 4 では、 `decision.kind` が `none` / `half` / `half_fallback` の場合に出社タグを付与するという境界を定義している。

ただ、実装上 `decision.kind == "none"` のときは通常勤務 (09:00-18:00) として登録されることになるので、この日もタグを付けるべきなんだけど、設計書に「`none` は通常勤務としてタグを付与する」という記述が明示されていない。

セクション 3 または 4 に「`decision.kind == "none"` (有給申請なし) の場合は通常勤務として扱い、出社タグを付与する」って一行追記すると、設計の完全性が上がるよ♡

対応内容:

    - 妥当と判断して、詳細設計の セクション 2 と セクション 4 に `none` の扱いを追記しました。
    - `none` は通常勤務としてタグ付与を継続する境界を明示し、`full` のみ除外対象であることを確定しました。
