# 009 メニュー選択 UI の左揃え修正設計

## このノートの位置づけ

- `001-このプロジェクトの概要書` の責務分離方針に従い、`menu` 層の描画責務だけを調整する
- 今回は設計のみを更新し、実装は行わない
- 操作仕様 (`↑` / `↓` / `Enter` / `Ctrl+C`) は既存どおり維持する

## 要望

選択肢 UI の表示が右側へずれて見えるため、選択中行・非選択行ともに左端起点で揃える。

現状イメージ:

```text
今回は何をしたい? (↑↓で選択 / Enterで決定 / Ctrl+Cで中断)
                                                         > 指定の月の平日に勤怠を詰め込む
                                                                                           指定月の平日勤怠をリセットする
```

期待する表示:

```text
今回は何をしたい? (↑↓で選択 / Enterで決定 / Ctrl+Cで中断)
> 指定の月の平日に勤怠を詰め込む
  指定月の平日勤怠をリセットする
```

## 対象範囲

- `app/menu/renderer.py` のメニュー再描画方式
- 左端基準の行描画ルール定義
- 既存 UI 操作仕様を維持したままの描画調整

## 非対象

- メニュー項目の増減・文言変更
- `app/menu/input_reader.py` のキー解釈仕様
- `app/menu/controller.py` の選択インデックス遷移仕様
- 認証・API・終了コード仕様

## 原因仮説（現行実装の根拠つき）

現行 `app/menu/renderer.py` の `render_menu` は、再描画時に次の処理をしている。

- `initial=False` のとき: `sys.stdout.write(f"\x1b[{len(items)}F")`
- 各行出力: `sys.stdout.write(f"{prefix} {item.label}\x1b[K\n")`

右寄りに見える主因は、各行末の `\n` が「改行（下方向移動）」だけで「復帰（先頭列へ戻る）」を保証しない端末があること。  
その場合、1行目のラベル末尾列から2行目を書き始めるため、行ごとに右へドリフトする。  
`\x1b[{len(items)}F` は再描画開始時の復帰には使っているが、行ごとの描画開始列は固定していないため、行単位ではずれが残る。

補足:

- この差は TTY 経路や端末実装差で顕在化しやすく、`docker compose run --rm app` と `docker compose exec app ...` で見え方が変わる可能性がある
- 端末幅を超える長いラベルで折り返しが発生すると、`len(items)` 行ぶんだけ戻す前提が崩れ、ずれが増幅しやすい

## 修正方針（レンダリング座標・カーソル制御）

### 1. 行の描画開始位置は `\r` で毎回 `column 1` に固定する（採用確定）

- 採用: `\r`
- 非採用: `\x1b[1G`
- 理由:
  - `\r` は ANSI CSI 依存がなく、キャリッジリターンとして広く安定
  - 現行でも ANSI を使うが、列復帰だけは最小制御文字で固定した方が端末差の影響を減らせる
  - 行ごとに `\r` を先頭へ入れることで、`"\n"` の端末依存差を吸収できる

### 2. 行クリアは `\x1b[K` を採用する（採用確定）

- 採用: `\x1b[K`（カーソル位置から行末までクリア）
- 非採用: `\x1b[2K`（行全体クリア）
- 理由:
  - 本修正では毎行 `\r` で先頭列から描画するため、残骸は「右側」だけ消せば十分
  - `\x1b[K` は現行実装でも使用しており、変更差分を最小化できる
  - `\x1b[2K` は行全体消去のため過剰で、再配置のための追加制御が不要な本件には利点が薄い

### 3. 再描画開始時に「メニュー先頭行」へ戻る手順を明確化する

- `initial=False` 時は、まず現在行先頭へ戻し、必要行数だけ上へ移動してメニューブロック先頭へ復帰する
- 「上移動だけ」「左移動だけ」に依存しないよう、縦移動と横位置リセットをセットで扱う

### 4. 選択行・非選択行の左端基準を統一する

- 選択行: `"> {label}"`
- 非選択行: `"  {label}"`
- どちらも 1 文字目から描画し、テキスト開始位置を揃える

### 5. 入力仕様は一切変更しない

- `↑` / `↓` / `Enter` / `Ctrl+C` の意味、選択遷移、終了コードマッピングは変更しない
- 変更は描画関数内の座標制御に限定する

### 6. 実装インターフェース案（設計で確定）

- `render_menu(items, selected_index, *, initial)` のシグネチャは変更しない
- `renderer.py` 内で以下の内部ヘルパーを追加する前提で実装する

```python
def _write_menu_line(*, selected: bool, label: str) -> None:
    ...
```

- 役割:
  - 行先頭へ復帰（`\r`）
  - `"> {label}"` / `"  {label}"` の描画
  - 行末クリア（`\x1b[K`）
  - 改行出力

## 確認観点（実装後チェックリスト）

1. ローカル実行 (`python main.py --auth-code ...`) で、初回表示から左端揃えになっている
2. `↑` / `↓` 押下ごとの再描画で、選択行・非選択行がともに左端から崩れない
3. `Enter` で従来どおり選択確定できる
4. `Ctrl+C` で従来どおり中断できる（終了コード `130`）
5. Docker `run` 経路（新規コンテナ）:
   - `docker compose run --rm app python main.py --auth-code <code>`
   - 表示が左揃えで、`↑` / `↓` 連打時も右ドリフトしない
6. Docker `exec` 経路（既存コンテナ）:
   - `docker compose up -d app` 後に `docker compose exec app python main.py --auth-code <code>`
   - `run` と同じ見え方になる
7. 非 TTY 条件の確認:
   - `docker compose run --rm -T app python main.py --auth-code <code>`
   - `docker compose exec -T app python main.py --auth-code <code>`
   - 期待どおりメニュー不可（環境エラー）で、描画崩れではなく仕様どおり終了する
8. 長い行の確認:
   - 端末幅を超える長さのラベル時に、折り返し後の再描画で左端起点が破綻しないこと
   - 少なくとも「右への累積ドリフトが増え続けない」ことを確認する
9. 複数入力の確認:
   - `↑↓` を 50 回以上連続入力しても、文字残り・右寄り表示・行ずれが発生しない
10. 項目数が多い場合の確認:
   - 現行仕様ではスクロール実装は対象外としつつ、表示行数が多くても描画処理が破綻しないことを確認する

## レビュー

### 1st review

1. **対象範囲の曖昧性**
    - `renderer.py` の「メニュー再描画方式」という表現が抽象的すぎる。具体的にどの関数・変数を変更するのか明示されていない。
    - `controller.py` が「必要時のみ」対象となっているが、「必要な場合」の判断基準が不明確。事前に `render_menu` のシグネチャ変更の有無を確定させるべき。

2. **原因仮説の根拠不足**
    - 「PTY 経路が変わると左端基準が崩れて見えるケースがある」と書いているが、実際に発生したログや再現手順の記載がない。仮説レベルで設計を進めると、修正後も問題が解決しない可能性がある。
    - カーソル復帰処理のどの部分が端末依存なのか、具体的なエスケープシーケンスとその挙動差を示すべき。

3. **修正方針の実装詳細不足**
    - `\r` と `\x1b[1G` の使い分け基準が示されていない。どちらを採用するのか、あるいは端末判定を行うのか明確にすべき。
    - 「行末は `\x1b[2K` または既存 `\x1b[K` でクリア」とあるが、両者の違い (行末までクリアか行全体クリアか) と選択理由が説明されていない。
    - `_write_menu_line` のようなヘルパー関数の「検討」レベルでは、実装時に設計意図が曖昧になる。関数シグネチャ案まで示すべき。

4. **既存実装の調査不足**
    - 現在の `app/menu/renderer.py` の実装内容が読まれていない前提で書かれている。現状のカーソル制御コードがどうなっているか、どの部分を変更するか具体的に記載されていない。
    - 「`render_menu(..., initial=...)` 呼び出し契約に変更が必要な場合」とあるが、現在の `render_menu` の実装を確認した上で、変更が必要かどうか判断すべき。

5. **確認観点の抜け**
    - 「複数回キー入力後も、文字の残り・右寄り表示・行ずれが発生しない」という項目はあるが、「長い選択肢が端末幅を超えた場合の挙動」についての考慮が抜けている。
    - メニュー項目が画面に収まらないほど多い場合 (スクロールが必要な場合) の挙動への言及がない。

6. **変更対象ファイル案の不完全性**
    - `app/menu/controller.py` の変更が「必要時のみ」となっているが、`render_menu` を呼び出す側の責務 (初期描画と再描画の区別) が明確でないと、`renderer.py` 側だけの修正で完結しない可能性が高い。
    - テストコードへの影響・追加が言及されていない。描画ロジック変更なら、少なくともユニットテストの追加・修正が必要なはず。

7. **非対象範囲の妥当性**
    - `app/menu/input_reader.py` を非対象としているが、入力待機中のカーソル位置と描画開始位置の整合性が取れているか確認が必要。描画だけ直しても入力待機時にズレる可能性がある。

8. **Docker 環境での検証計画の具体性不足**
    - 「Docker 経路を含めて左揃えが再現性高く維持されるか」とあるが、どの Docker 構成 (TTY あり/なし、stdin/stdout のパイプ有無) で確認するか明示されていない。

4. 既存実装の調査不足: 現行 `renderer.py` の挙動を前提に、どの処理を置き換えるかを設計本文へ追記した。
5. 確認観点の抜け: 長い行（端末幅超過）と連続入力（50回以上）に加え、項目数が多い場合の扱い（非対象だが破綻しない確認）を追記した。

7. 非対象範囲の妥当性: 入力仕様・入力解釈は非変更としたうえで、非TTY検証を確認観点へ追加した。
8. Docker 検証計画の具体性不足: `docker compose run --rm app` と `docker compose exec app ...`（TTY/非TTY）を具体コマンドで明記した。

判定: LGTM 依頼

### 2nd review

1. **原因仮説の端末環境特定度**
    - 「`\n` が復帰を保証しない端末がある」という説明は一般論として妥当。Docker 経路差（`run` / `exec`）での見え方については、再現コマンドと観測結果を残しておくと保守性が上がる。

2. **`\x1b[K` の役割明確化**
    - ラベル長が短くなったときの残骸も `\x1b[K` で処理する旨を明示しておくと、意図がさらに明確になる。

3. **`_write_menu_line` の役割と再利用性**
    - 将来の装飾拡張（記号・色付け）に備えた補足コメントがあると拡張しやすいが、本件の範囲外として現状でも問題なし。

4. **長い行の折り返し時の制約**
    - `\x1b[{len(items)}F` は論理行基準のため、折り返しで物理行が増えると復帰位置が崩れる可能性がある。長いラベルは非推奨として制約明記があるとより安全。

5. **非 TTY 条件の挙動**
    - `-T` 経路での期待終了コード（`2`）を確認観点に入れると、終了コード設計との一貫性が高まる。

6. **実装後差分の記録運用**
    - 実装時に想定外差分が出たら設計書へ追記する運用を明示すると、設計と実装の乖離防止に有効。

判定: LGTM

- `app/menu/renderer.py` に `_write_menu_line(selected: bool, label: str)` を追加した。
