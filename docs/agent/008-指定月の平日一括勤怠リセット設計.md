# 008

> [!WARNING]
> この設計は対象日判定（`day_pattern` / `use_default_work_pattern`）に誤りを含むため obsolete 扱い。
> リセット機能の正本は `docs/agent/010-勤怠リセットを全日対象へ修正設計.md` を参照すること。

## 要望

新しい機能「指定月の平日勤怠をリセットする」を追加する。

- 目的: テストのため、指定月の登録状況をリセットする
- 処理対象の判定条件は 006 + 007 と同じ
  - `day_pattern == "normal_day"`
  - `use_default_work_pattern == true`
- 上記条件を満たす日付に対して、以下を実行する
  - 勤怠を空にする
  - 勤怠タグを空にする

### 勤怠を空にする API

```http
PUT /hr/api/v1/employees/{employee_id}/work_records/{date}
{
  "company_id": {company_id},
  "break_records": [],
  "work_record_segments": []
}
```

### 勤怠タグを空にする API

```http
PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}
{
  "company_id": {company_id},
  "employee_attendance_tags": []
}
```

## 詳細設計

### 1. 設計ゴール

指定月 (`yyyy-mm`) の日付を 1 日から末日まで走査し、`006 + 007` と同じ実行対象条件を満たす日に対して、以下 2 API を順に実行して勤怠登録状態を初期化する。

- `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}` に `break_records: []`, `work_record_segments: []`
- `PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}` に `employee_attendance_tags: []`

用途はテスト用データの月次リセットであり、本番運用向けの業務フロー変更は目的に含めない。

### 2. メニュー文言（統一）

メニュー表示名は以下に固定する。

- `指定月の平日勤怠をリセットする`

### 3. API endpoint と `company_id` 受け渡し統一方針

#### API endpoint（完全パス）

- `GET /hr/api/v1/users/me`
- `GET /hr/api/v1/employees/{employee_id}/work_records/{date}?company_id={company_id}`
- `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}`
- `PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}`

#### `company_id` 受け渡し方針

- GET (`work_records`) は query に `company_id` を付与する
- PUT (`work_records`, `attendance_tags`) は body に `company_id` を付与する
- PUT では query に `company_id` を付けない運用で統一する

### 4. 処理フロー（007 の判定順を本書で固定）

```text
1. メニューから「指定月の平日勤怠をリセットする」を選択
2. input() で対象月 yyyy-mm を受け取り、妥当性を検証
3. GET /hr/api/v1/users/me で company_id, employee_id を取得
4. 対象月の日付リスト (1日〜末日) を生成
5. 日付ループ
   5-1. GET /hr/api/v1/employees/{employee_id}/work_records/{date}?company_id=... で判定用項目を取得
   5-2. day_pattern != normal_day なら SKIP (reason=day_pattern_not_normal_day)
   5-3. use_default_work_pattern != true なら SKIP (reason=use_default_work_pattern_false)
   5-4. PUT /hr/api/v1/employees/{employee_id}/work_records/{date} (body.company_id + 空配列 payload)
   5-5. PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date} (body.company_id + 空配列 payload)
   5-6. 日次ログ出力 (OK)
   5-7. 共通定数秒数だけ待機
6. エラーがなければ完了サマリー出力
7. 途中エラー時はサマリーを出して即中断
```

判定順は 007 依存の参照だけにせず、本設計内で次の順に固定する。

- 1段目: `day_pattern == "normal_day"` 判定
- 2段目: `use_default_work_pattern == true` 判定

### 5. 既存 `bulk_attendance` との共通化/差分方針

#### 共通化方針（シグネチャ案）

`app/actions/bulk_attendance_common.py` を追加し、以下を共通関数として定義する案を採用する。

- `def parse_target_month(raw: str) -> tuple[int, int] | None`
- `def generate_dates(year: int, month: int) -> list[str]`
- `def resolve_user_ids(hr_client: HrApiClient) -> tuple[int, int]`
- `def evaluate_target_day(work_record_body: dict[str, object]) -> tuple[bool, str | None, str | None]`
  - 戻り値: `(is_target, reason_code, detail)`
- `def format_api_error(exc: ApiResponseError) -> str`
- `def print_summary(ok_count: int, skip_count: int, error_date: str | None) -> None`

#### 差分方針

差分は「対象日に何を PUT するか」のみを基本とする。

- 既存 `bulk_attendance`:
  - 勤怠を `09:00-19:00` + 休憩 `11:30-12:30` で登録
  - 「出社」タグ ID を取得して付与
- 新規 `bulk_attendance_reset`:
  - 勤怠 payload を空配列で上書き
  - 勤怠タグ payload を空配列で上書き
  - 出社タグ ID の取得処理は不要

#### 段階導入方針（設計時点で固定）

- フェーズ1: `bulk_attendance_reset.py` を追加し、`bulk_attendance.py` には機能変更を入れない
- フェーズ2: 動作確認後、上記シグネチャで `bulk_attendance_common.py` へ抽出し、`bulk_attendance.py` と `bulk_attendance_reset.py` の双方を置換
- フェーズ3: 006/008 の設計ノートに「共通化完了」を追記して仕様を同期

今回は設計更新のみであり、フェーズ実行は実装タスクで行う。

### 7. ログ仕様（OK/SKIP/ERR）

#### OK

- 形式: `[OK]   <date> 勤怠/タグを空へ更新済み`
- 例: `[OK]   2026-03-10 勤怠/タグを空へ更新済み`

#### SKIP

007 と同じ理由コードを使う。

- `day_pattern` 不一致:
  - `[SKIP] <date> reason=day_pattern_not_normal_day detail=day_pattern=<value>`
- `use_default_work_pattern` 不一致:
  - `[SKIP] <date> reason=use_default_work_pattern_false detail=use_default_work_pattern=<value>`

1 日につき理由は 1 つだけ出力する（判定順は `day_pattern` → `use_default_work_pattern`）。

#### ERR

- 形式: `[ERR]  <date> <prefix>: <api_error_messages>`
- 例: `[ERR]  2026-03-15 勤怠リセット失敗: 給与明細が確定されているので、更新できません。`

`ApiResponseError.response_body.errors[].messages` が取得できる場合は結合して表示し、取得できない場合は例外文字列を表示する。

### 8. エラー時中断方針（ロールバック非実施を明記）

- 中断トリガー:
  - `GET /hr/api/v1/employees/{employee_id}/work_records/{date}` 失敗
  - `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}` 失敗
  - `PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date}` 失敗
- 中断動作:
  - 当日処理を `ERR` 出力
  - サマリーを `中断: <ok>日成功 / <skip>日スキップ / 1日エラー (<date> で中断)` 形式で出力
  - `EXIT_CODE_APP_ERROR (1)` で終了
- 継続条件:
  - `SKIP` は正常系として次日に進む
- 正常完了:
  - `完了: <ok>日成功 / <skip>日スキップ / 0日エラー`
  - `EXIT_CODE_OK (0)` で終了

#### `work_records` 成功後に `attendance_tags` が失敗した場合

- 扱い: ロールバックせず、その時点で中断する
- 理由:
  - `work_records` と `attendance_tags` は別 endpoint であり、跨りトランザクションは提供されない
  - 疑似ロールバックとして元データ復元を試みると、失敗連鎖で状態をさらに悪化させるリスクがある
  - 本機能はテスト用途で、再実行により空状態へ収束させる運用が可能（PUT は同一 payload で冪等）
- 運用:
  - `[ERR]` 出力後に即中断
  - オペレータはエラー解消後に同月を再実行する

## レビュー

### 1st review

#### 指摘事項

1. **API endpoint 表記の不整合**
    - 5-1 は `GET /work_records/{date}?company_id=...` だが、セクション 7 の payload には `company_id` が含まれている。 freee API 実装は通常 query param と body param で company_id を両方要求するので、 PUT 時の query param 記述漏れを確認すること。
    - セクション 1 に記載の API endpoint は省略形。最低限 `/hr/api/v1/employees/{employee_id}/work_records/{date}` と `/hr/api/v1/employees/{employee_id}/attendance_tags/{date}` を明記すべき。

2. **処理順序とエラー中断の厳密性**
    - 5-4 で PUT work_records → 5-5 で PUT attendance_tags の順で失敗した場合、勤怠だけ空になりタグが残る。中途半端な状態になるが運用上許容可能かを明示する必要がある (テスト用途とはいえ要確認)。
    - 「途中エラー時は即中断」は明記されているが、 rollback 不要の理由を補足すべき。

3. **共通化の実装方針が曖昧**
    - セクション 3 に「実装時に共通ヘルパーへ切り出す」とあるが、「設計のみで実装しない」と矛盾している。設計段階で共通化関数のシグネチャ案を示すか、実装タイミングを再定義すべき。

4. **SKIP 理由の判定順が 007 と合っているか未検証**
    - 007 の判定順を本設計書内で再確認していない。 007 を参照せよとあるだけでは、 007 を更新したときに 008 が追従漏れになる可能性がある。

5. **待機時間のパラメータ化不要性**
    - 5-7 で 0.5 秒待機をハードコードしているが、今後 006, 007, 008 が同じ値を持つならプロジェクト共通定数にすべき。設計書にその検討を含めるべき。

6. **メニュー項目名の日本語表記**
    - セクション 2 では「指定月の平日勤怠をリセットする」、セクション 1 では「指定の月の平日をぜんぶ勤怠空っぽにする」と表記揺れ。統一が必要。

7. **サマリー出力の表記**
    - セクション 6 で `中断: <ok>日成功 / <skip>日スキップ / 1日エラー (<date> で中断)` とあるが、 `<ok>` の具体的な埋め込み形式が不明 (変数名 or プレースホルダ)。実装例を示すべき。
