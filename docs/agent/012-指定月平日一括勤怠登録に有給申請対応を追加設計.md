### holiday_type:full について有給の申請がある日については、勤怠の登録方法をこうする

```bash
curl --location --request PUT 'https://api.freee.co.jp/hr/api/v1/employees/<employee_id>/work_records/<date>-12' \
--header 'Accept: application/json' \
--header 'Authorization: Bearer <token>' \
--data '{
  "company_id": <company_id>,
  "paid_holidays": [
    {
      "type": "full"
    }
  ]
}'
```

### holiday_type:half について有給の申請がある日については、勤怠の登録方法をこうする

```bash
{
  "company_id": {{FREEE_COMPANY_ID}},
  "work_record_segments": [
    {
      // デフォルトである 09:00 - 19:00 から、 half で申請されている時間を引いたものを登録する。もし 09:00 - 19:00 をふたつに分けるような変な申請だった場合は、デフォルトで登録して、ログに警告コメントを出す。
      "clock_in_at": "2026-03-31 14:00:00",
      "clock_out_at": "2026-03-31 19:00:00"
    }
  ],
  "paid_holidays": [
    {
      "type": "half",
      "mins": 240
    }
  ]
}
```

### 所感

- というわけで、これまではこう↓だったが……
    - 平日 -> 09:00 - 19:00 で勤怠登録
    - 休日 -> 勤怠登録なし
- これからはこう↓になる。
    - 平日で有給申請なし -> 09:00 - 19:00 で勤怠登録
    - 平日で有給申請あり(holiday_type:full) -> 有給として勤怠登録
    - 平日で有給申請あり(holiday_type:half) -> 半休として勤怠登録
    - 休日 -> 勤怠登録なし

## 詳細設計

### 1. この設計で追加すること

既存の「指定月の平日一括勤怠登録」に対して、対象月の有給申請 (`paid_holidays`) を先に取り込み、日付ごとに以下の分岐で `work_records` の PUT payload を切り替える。

- 平日かつ有給なし: 既存どおり `09:00-19:00` 登録
- 平日かつ `holiday_type=full`: `paid_holidays=[{"type":"full"}]` で登録
- 平日かつ `holiday_type=half`: `09:00-19:00` から申請時間を差し引いた勤務帯 + `paid_holidays=[{"type":"half","mins":...}]` で登録
- 休日: 既存どおりスキップ

### 2. API 呼び出し順

1. 対象月 (`yyyy-mm`) を入力
2. `GET /hr/api/v1/users/me` で `company_id` と `employee_id` を確定
3. `GET /hr/api/v1/approval_requests/paid_holidays` で対象月の有給申請を全件取得
4. 対象月の日付を 1 日ずつ処理
5. 各日で `GET /hr/api/v1/employees/{employee_id}/work_records/{date}` を取得して対象日判定
6. 対象日のみ `PUT /hr/api/v1/employees/{employee_id}/work_records/{date}` を実行

補足:

- `paid_holidays` は「処理開始時」に 1 回まとめて取得し、`target_date` をキーに辞書化して日次処理で再利用する
- `paid_holidays` 取得は `limit=100` 固定、`offset` を `0 -> 100 -> 200 ...` で増やしてページングする
- ループ条件は `取得済み件数 >= total_count`。`total_count` 未達なのに応答件数が 0 の場合は異常として即中断する

### 3. 対象日の判定ルール (平日/休日)

対象日の最終判定はカレンダー曜日ではなく、`work_records/{date}` のレスポンス値を正とする。

- 実行対象条件:
  - `day_pattern == "normal_day"`
  - `use_default_work_pattern == true`
- 上記どちらかを満たさない日はスキップ

スキップログ例:

- `[SKIP] 2026-03-01 reason=day_pattern_not_normal_day detail=day_pattern=prescribed_holiday`
- `[SKIP] 2026-03-10 reason=use_default_work_pattern_false detail=use_default_work_pattern=false`

### 4. 有給申請データの採用ルール

1 日につき利用する有給申請は 0 件または 1 件に正規化する。

- 採用対象:
  - `target_date` が当日一致
  - `revoke_status == null`
  - `status` が `approved` または `in_progress`
  - `holiday_type` が `full` または `half`
- 優先順位:
  - `full` が 1 件以上ある場合は `full` を採用
  - `full` が複数ある場合は `issue_date` が新しいものを優先し、同値なら `id` が大きいものを採用して `WARN` を出す
  - `full` がなく `half` が 1 件なら `half` 採用
  - 上記以外 (同日複数 `half`、不正データ混在、status 非採用など) はフォールバック扱い

`status` 非採用データ (`rejected` など) は勤怠入力には使わず、`[INFO]` で除外理由だけ出力する。

### 5. half の時間差分ロジック

基準勤務帯は固定で `09:00-19:00`。
`half` の `start_at` / `end_at` を当日日時へ展開し、差分を 1 本の `work_record_segments` に落とす。

- 午前半休パターン:
  - 条件: `start_at == 09:00` かつ `09:00 < end_at <= 19:00`
  - 休暇区間: `09:00` から `end_at` まで
  - 勤務区間: `end_at` から `19:00` まで
  - `mins`: `minutes_between(09:00, end_at)`
- 午後半休パターン:
  - 条件: `end_at == 19:00` かつ `09:00 <= start_at < 19:00`
  - 勤務区間: `09:00` から `start_at` まで
  - 休暇区間: `start_at` から `19:00` まで
  - `mins`: `minutes_between(start_at, 19:00)`

`break_records` の現時点仕様:

- 有給なしの日: 既存どおり `11:30-12:30` を送る
- `holiday_type=full`: `break_records` は送らない
- `holiday_type=half`: `break_records` は送らない

`half` 採用時の payload 例:

```json
{
  "company_id": 123,
  "work_record_segments": [
    {
      "clock_in_at": "2026-03-31 14:00:00",
      "clock_out_at": "2026-03-31 19:00:00"
    }
  ],
  "paid_holidays": [
    {
      "type": "half",
      "mins": 300
    }
  ]
}
```

### 6. フォールバック条件

次のいずれかに該当した `half` は「変則 half」とみなし、有給連携を使わず通常勤務登録へフォールバックする。

- `start_at` / `end_at` が null
- `start_at >= end_at`
- 申請時間が `09:00-19:00` の外に出る
- 申請時間が勤務帯の中央にあり、差分が 2 セグメント必要になる
- 同日に `half` が複数あり一意に決められない

中央分断 half の機械判定式:

- `is_center_split = (09:00 < start_at) and (end_at < 19:00) and (start_at < end_at)`
- `is_center_split == true` のときはフォールバック
- 端一致 (`start_at == 09:00` または `end_at == 19:00`) は中央分断ではない

フォールバック時の動き:

- payload は「有給なし」の既定 (`09:00-19:00`) を使用
- `paid_holidays` は送らない
- 警告ログを出す

警告ログ例:

- `[WARN] 2026-03-31 reason=half_fallback detail=request_id=16080378,start_at=10:00,end_at=15:00`

### 7. エラー時方針

- 月初の前処理 (`users/me`, `paid_holidays` 取得) が失敗したら、その時点で処理中断して終了コード `1`
- 日次処理中に `work_records` の GET/PUT が失敗したら、当日エラーを出して即中断 (既存方針を維持)
- 異常系ログは必ず API ステータスコードとレスポンスの `errors[].messages[]` を出力
- `half` データ不整合は API エラーではないためフォールバックで継続する

継続可否の最終方針:

- API 通信エラー: 即中断 (採用)
- データ不整合 (`half_fallback`): 警告を出して継続 (採用)
- 「失敗日をスキップして API エラーでも続行するモード」: 今回は非採用。理由は「誤登録の見逃し防止」と「既存運用との整合」。

エラーログ例:

- `[ERR] 2026-03-12 step=put_work_record status=400 message=給与明細が確定されているので、更新できません。`

### 8. 将来拡張ポイント

- 変則 `half` を 2 セグメント勤務として正しく登録する拡張
- `status` 採用条件 (`approved` / `in_progress`) の設定値化
- 半休時の休憩 (`break_records`) 自動計算ルールを追加
- 有給日 (`full`/`half`) のときの勤怠タグ付与ルールを明示的に分岐

### 9. 想定実装ファイルと責務

  - 月次処理フロー制御
  - 日付ごとの有給適用判定

  - ログ出力 (`OK/SKIP/WARN/ERR`) と中断判定
- `app/clients/hr_api_client.py`

  - 有給判定で使う定数 (`09:00`, `19:00`, `paid_holidays limit=100`) の集約先

### 10. テスト方針 (最低限)

- 正常系:
  - 平日・有給なし -> 通常 payload (`09:00-19:00`, 休憩あり)
  - 平日・`full` (approved) -> `paid_holidays=[{\"type\":\"full\"}]`
  - 平日・`half` 午前 (`09:00-14:00`) -> 勤務 `14:00-19:00`, `mins=300`
  - 平日・`half` 午後 (`13:00-19:00`) -> 勤務 `09:00-13:00`, `mins=360`
- 準正常/フォールバック:
  - `half` 中央分断 (`10:00-15:00`) -> `half_fallback` で通常 payload
  - 同日複数 `half` -> `half_fallback` で通常 payload
  - `status` が非採用 (`rejected` など) -> 有給なし扱い
- 異常系:
  - `paid_holidays` ページング途中で 0 件応答かつ `total_count` 未達 -> 即中断
  - 日次 `GET work_records` 失敗 -> 即中断
  - 日次 `PUT work_records` 失敗 -> 即中断

## レビュー

- 5. half の時間差分ロジックで、 mins の計算が逆になってる
    - 「午前半休パターン」で `mins: end_at - 09:00` は「休んだ時間」だけど、これは `paid_holidays` の `mins` に入れる値だから正しい
    - 「午後半休パターン」で `mins: 19:00 - start_at` も同じく「休んだ時間」なので正しい
    - ただし、 payload 例では `clock_in_at: 14:00` ~ `clock_out_at: 19:00` で `mins: 300` (5 時間分) になってるけど、これは `09:00 - 14:00` の休憩 (5 時間 = 300 分) を示すので計算は合ってる
    - でもロジック説明の書き方がわかりにくい。「勤務帯: `end_at - 19:00`」って書くと引き算みたいに見える。「勤務帯: `end_at` から `19:00` まで」とか書いたほうが明確
    - 対応内容: 採用。計算ロジック自体は維持し、詳細設計 5 を「休暇区間/勤務区間」の明示表現へ書き換えて引き算表記を廃止した。
- 4. 有給申請データの採用ルールに `status` の条件が抜けてる
    - `revoke_status == null` はチェックしてるけど、 `status` が `approved` や `in_progress` とかどれを採用するか明記されてない
    - 要望の JSON 例では `status: "in_progress"` と `status: "approved"` が両方あるから、どっちも使うのか、片方だけ使うのか曖昧
    - 8. 将来拡張ポイントで「設定値化」って書いてるけど、現時点の挙動も決めないとまずい
    - 対応内容: 採用。現時点の採用条件を `status in {approved, in_progress}` に確定し、非採用 status は除外ログを出す方針へ明記した。
- 6. フォールバック条件で「中央にある half」の判定ロジックが不明
    - 「申請時間が勤務帯の中央にあり」って書いてるけど、具体的にどう判定するのか説明がない
    - 例えば `09:00-12:00` の half は午前だから OK だけど、 `10:00-15:00` みたいなのは NG なのかな?
    - この判定ロジックを明確にしないと実装者が困る
    - 対応内容: 採用。`is_center_split = (09:00 < start_at) and (end_at < 19:00) and (start_at < end_at)` を機械判定式として追加し、true ならフォールバックに確定した。
- 2. API 呼び出し順で `paid_holidays` の取得タイミングが曖昧
    - 「月初に 1 回取得」って書いてるけど、手順 2 と 3 の間で取得するってことだよね
    - 「月初」って言葉は「対象月の最初の日」じゃなくて「処理の最初のほう」って意味だと思うけど紛らわしい
    - あと `limit` を固定してページング処理するって書いてるけど、 `limit` は何件にするのか書いてない
    - 対応内容: 採用。「処理開始時に 1 回取得」へ文言修正し、`limit=100` 固定・`offset` 増分ページング・`total_count` 未達で 0 件応答なら異常中断まで具体化した。
- 7. エラー時方針で「日次処理中の失敗」が即中断なのは仕様として厳しすぎないか?
    - 30 日間の勤怠登録を回してるときに、 15 日目で 1 件失敗しただけで全部止まるのは UX が悪い
    - 既存方針を維持するって書いてるけど、有給対応を入れる今回のタイミングで改善したほうがよくない?
    - 少なくとも「失敗した日をスキップして続行する」オプションは検討すべき
    - 対応内容: 非採用。API エラーは誤登録見逃しリスクを避けるため即中断を維持し、継続は `half` データ不整合のフォールバック時のみ許可する方針に確定した。
- 5. half の時間差分ロジックで「休憩時間」の扱いが不明
    - `09:00-19:00` は 10 時間勤務だけど、実際は休憩 1 時間あるよね? (普通は)
    - half の mins 計算で休憩を考慮するのか、しないのか明記されてない
    - 8. 将来拡張ポイントに `break_records` 自動計算って書いてるけど、現時点の仕様も決めないとダメ
    - 対応内容: 採用。現時点仕様として「有給なしは既存どおり休憩あり、full/half は `break_records` を送らない」を明記し、半休時の休憩自動計算は将来拡張へ分離した。
- 実装パスが書かれてない
    - 概要書によれば、新機能は `app/actions/<action_name>.py` に実装して `registry` に登録する流れだけど、この設計書にはその指示がない
    - 既存の「指定月の平日一括勤怠登録」がどこにあるのかも不明
    - `app/clients/` に追加する API クライアント関数名も明記すべき
    - 対応内容: 採用。詳細設計 9 を追加し、`app/actions/bulk_attendance.py`・`app/clients/hr_api_client.py`・`app/actions/bulk_attendance_common.py` の責務を明記した。
- テスト方針が書かれてない
    - 有給の `status` や `holiday_type` のバリエーション、フォールバック条件、エラーパターンをどうテストするのか記載なし
    - Postman 動作確認を書くって言ってるけど、どのパターンを確認するのか列挙すべき
    - 対応内容: 採用。詳細設計 10 を追加し、正常系・フォールバック系・異常系の最低限ケースを列挙した。
