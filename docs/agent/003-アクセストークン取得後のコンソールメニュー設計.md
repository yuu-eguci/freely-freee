# 003 アクセストークン取得後のコンソールメニュー設計

## このノートの位置づけ

- 本ノートは LGTM 前の設計更新版
- 002 ノートの認証フローが成功した直後の UX を、実装者が迷わない粒度で固定する
- オーナー指摘の「番号入力ではなく上下カーソル選択」を設計の中心に据える

## 要望

- OAuth でアクセストークン取得に成功した直後に、次の行動を対話 UI で選べるようにする
- 選択 UI は番号入力ではなく、`↑/↓` と `Enter` で操作する
- 今回の選択肢は 1 件のみ
  - `あ、いや、アクセストークン取得までいけるか見たかっただけ`
- この選択肢を確定したら、検証用としてアクセストークンを標準出力に表示して終了する

## 目的

- 「トークン取得まで動くことの確認」を、実行時に明示的に完了できるようにする
- 今後の勤怠機能追加に向けて、共通メニュー基盤 (表示・選択・実行) を先に作る

## 対象範囲

- 対象: `main.py` の成功パス (認可コード交換成功時 / リフレッシュ成功時) の直後に呼ばれるメニュー処理
- 非対象: 勤怠 API の実処理、複数機能の実装

## 仕様

### 1. メニュー表示条件

- `access_token` の取得に成功したときは必ずメニューを表示する
  - 認可コードフロー成功時
  - リフレッシュフロー成功時

### 2. UI 仕様 (上下カーソル選択)

- 表示形式は以下を基準とする

```text
今回は何をしたい？（↑↓で選択 / Enterで決定 / Ctrl+Cで中断）
> あ、いや、アクセストークン取得までいけるか見たかっただけ
```

- `>` は現在選択中の行を示す
- 今回は選択肢が 1 件のみなので、`↑/↓` を押しても選択位置は変化しない
- 次回以降に選択肢が増えた場合も、同じ UI コンポーネントを使って選択位置だけ変える

### 3. 入力イベント仕様

- `↑` (`\x1b[A`): 1 つ上へ移動
  - 先頭要素で押した場合は先頭に留まる (循環させない)
- `↓` (`\x1b[B`): 1 つ下へ移動
  - 末尾要素で押した場合は末尾に留まる (循環させない)
- `Enter` (`\r` / `\n`): 現在選択中のアクションを確定
- `Ctrl+C` (`KeyboardInterrupt`): ユーザー中断として終了コード `130`
- `Ctrl+D` / EOF: 入力異常として `MenuInputError` (終了コード `2`)
- 上記以外のキー: 無視して入力待ち継続 (エラー終了しない)

### 4. トークン表示仕様

- 選択肢 `あ、いや、アクセストークン取得までいけるか見たかっただけ` が確定した場合のみ、以下を標準出力へ 1 回表示

```text
access_token: {取得したアクセストークン}
```

- 本出力は「手元で OAuth 到達確認をする検証フェーズ」に限定して許可する
- このフェーズではログファイル出力機構を持たず、明示選択時のみ出力する
- 将来方針: 本番利用に寄せる段階で「デフォルトはマスク表示 + 明示確認時のみ全表示」に切り替える

### 5. 終了コード仕様

- `0`: 正常終了 (メニューでアクション確定し、処理完了)
- `1`: 既存の認証/設定/保存系エラー (`ConfigError`, `TokenStoreError`, `OAuthTokenError`)
- `2`: メニュー入力・端末条件エラー (`MenuInputError`, `MenuEnvironmentError`)
- `130`: ユーザー中断 (`Ctrl+C`)

## 例外方針

- 既存方針 (`ConfigError`, `TokenStoreError`, `OAuthTokenError`) は維持する
- メニュー層の例外を追加して責務を分離する
  - `MenuError` (基底)
  - `MenuEnvironmentError`: `stdin/stdout` が TTY ではなく、カーソル UI を提供できない
  - `MenuInputError`: EOF などで選択確定不能
  - `MenuCancelled`: 必要なら `KeyboardInterrupt` をラップして扱う
- `main()` は「例外を終了コードへ変換する責務」のみ持つ

## 責務分離

- `main()`:
  - 認証フロー実行
  - `access_token` 取得後にメニュー層を呼び出す
  - 例外を終了コードへマッピングして `SystemExit(code)` で終了
- `menu/controller`:
  - 画面表示ループ、キー入力解釈、選択確定
  - ビジネス処理は直接実行しない
- `menu/actions`:
  - 選択確定後の処理本体を実行
  - 今回は `print_access_token(access_token: str) -> int` のみ実装

実装時の関数シグネチャは次を基準とする。

```python
def run_post_token_menu(access_token: str) -> int: ...
def select_menu_action(items: list["MenuItem"]) -> "MenuItem": ...
def execute_menu_action(action: "MenuAction", access_token: str) -> int: ...
```

## 実行フロー (確定版)

1. `main()` で認証フロー成功
2. `access_token` を取得
3. `exit_code = run_post_token_menu(access_token)`
4. `raise SystemExit(exit_code)`

`main()` は「戻り値を返して終わり」ではなく、最終的に必ず `SystemExit` へ接続する。

## 想定実行例

```bash
$ python main.py --auth-code xxxxx
認可コードでトークンを取得し、token.json を更新しました。
今回は何をしたい？（↑↓で選択 / Enterで決定 / Ctrl+Cで中断）
> あ、いや、アクセストークン取得までいけるか見たかっただけ
access_token: eyJhbGciOi...
```

## 次段階への接続

- 選択肢追加時は `MenuItem` を増やすだけで UI 層はそのまま利用する
- 例: `勤怠を記録する`, `今日の勤怠を確認する` を追加可能
- 今回は選択肢 1 件のみ実装し、共通基盤を先に固める

## レビュー対応結果 (LGTM 前)

- [修正必須] メニューを番号入力にしているのが弱いです。オーナーレビューとして必ず指摘しますが、番号入力ではなく、上下カーソルで選択できる UI にしてください。1 件しか選択肢がない現状でも、拡張前提の導線としてはこちらのほうが自然です。  
  対応内容: 番号入力仕様を廃止し、`↑/↓ + Enter` のイベント仕様・表示仕様へ全面置換した。選択肢 1 件時の挙動と、将来複数件時の拡張方式も明記した。

- [修正必須] `select_action(access_token: str) -> int` としているのに、`main()` でその戻り値をどうプロセス終了コードへ反映するのかが曖昧です。単に値を返すだけでは終了コードは変わらないので、`sys.exit(...)` まで含めて責務を明記してください。  
  対応内容: `main()` の責務を「終了コードマッピング + `SystemExit`」まで固定し、`run_post_token_menu` 呼び出し後に `raise SystemExit(exit_code)` する実行フローを確定した。

- [修正必須] `access_token` をそのまま標準出力へ出す設計は、ログやリダイレクト先に残りやすく危険です。今回が検証用途だとしても、少なくとも「なぜ生出力してよいのか」の前提と、将来はマスクや確認表示に切り替える方針を明記しないと危ういです。  
  対応内容: 生出力を「手元検証フェーズ限定」であること、明示選択時のみ 1 回出力すること、将来はマスク表示へ切り替える方針を仕様に追加した。

- [修正必須] 入力エラーの扱いが雑です。「空文字も不正」と書くだけだと、空白のみ、改行のみ、EOF、Ctrl-C などの扱いが実装依存になります。どこまでを不正入力として扱い、どの例外や終了コードに落とすのかをもう一段具体化してください。  
  対応内容: `↑/↓/Enter/Ctrl+C/EOF/その他キー` ごとの挙動を定義し、`2` (入力異常) と `130` (ユーザー中断) を分離した。あわせて `MenuInputError`, `MenuEnvironmentError`, `MenuCancelled` を定義した。

- [修正必須] 現状は選択肢が 1 つしかないのに、わざわざメニューを挟む理由が薄いです。拡張ポイントを残す意図は分かりますが、それなら「将来の複数機能を見据えた共通メニュー基盤」として、メニュー表示・選択・実行の責務分離まで書かないと設計として中途半端です。  
  対応内容: `main()` / `menu/controller` / `menu/actions` の 3 層に責務分離し、関数シグネチャまで固定した。将来の選択肢追加を `MenuItem` 拡張で吸収する設計に更新した。
