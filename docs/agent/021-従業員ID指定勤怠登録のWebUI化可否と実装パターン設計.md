# 021 従業員ID指定勤怠登録のWebUI化可否と実装パターン設計

## 要望

- 「指定の月に従業員ID指定で勤怠を詰め込む」機能を Web インターフェース化したい。
- 起動イメージは `docker compose run --rm app pipenv run python web.py`。
- localhost で操作できる画面にしたい。
- token.json 更新まではこれまで通りの流れを維持したい。
- フォームで入力したい項目:
  - 対象月
  - 出勤何時
  - 退勤何時
  - 従業員ID
  - 出社タグの有無
- 送信後は処理中UI（ぐるぐる）と待機メッセージを表示したい。

## このノートで決める範囲

- 今回は設計のみ（実装・コード変更は次タスク）。
- 可否判断、実装パターン比較、運用方式、入力仕様、UI/セキュリティ要件までを確定する。

## 詳細設計

### 1) 可否判断（技術的に可能か、前提条件）

- 結論: **技術的に実現可能**。
- 既存資産として `app/actions/bulk_attendance.py` に「従業員ID指定の月次一括処理」本体があり、Web は入力手段を `input()` から HTTP フォームへ置き換えるのが主な差分。
- 前提条件:
  - `token.json` が有効な形式で配置されていること（`refresh_token` 必須）。
  - freee 側で対象従業員に対する API 実行権限があること。
  - `docker compose run` で起動する場合、localhost へポート公開設定をすること（後述）。

### 2) 実装パターン比較（最低2案。推奨案を1つ）

- 案A: **同期実行型（推奨）**
  - 概要: `web.py` の POST でそのまま処理を実行し、完了後に結果を返す。
  - メリット:
    - 構成がシンプルで実装コストが低い。
    - 既存の日次ループ/ログ規約をそのまま流用しやすい。
    - 依存追加なしでも成立しやすい。
  - デメリット:
    - 処理中は同一リクエストを占有する。
    - 同時実行制御をしないと二重投入リスクがある。

- 案B: 非同期ジョブ型
  - 概要: フォーム送信でジョブを積み、別ワーカーで実行。画面はポーリングで進捗表示。
  - メリット:
    - 長時間処理でも UI 応答性を確保しやすい。
    - 実行履歴・再実行制御を作り込みやすい。
  - デメリット:
    - ジョブ管理ストア、状態遷移、失敗時復旧が必要で設計が重い。
    - 今回スコープ（まず Web 化）に対して過剰になりやすい。

- 推奨案: **案A（同期実行型）**
  - 理由: まずは「CLI の既存ロジックを壊さずに Web 入力へ置換」が目的で、最短で安全に進められるため。

### 3) `web.py` 起動方式（docker compose run）での現実的運用

- `compose.yml` に `ports:` 定義がないため、下記どちらかが必要。
  - `docker compose run --rm -p 127.0.0.1:8000:8000 app pipenv run python web.py --host 0.0.0.0 --port 8000`
  - もしくは将来 `compose.yml` に `ports` を追加したうえで `--service-ports` を使う。
- 運用の実態としては、**1人運用・単発起動**が前提。
  - 作業時だけ `run --rm` で起動。
  - 処理終了後は `Ctrl+C` で停止し、コンテナは自動削除。
- `token.json` は bind mount されるため、コンテナ内更新がホストへ反映される前提で運用可能。

### 4) token.json 更新まで既存フロー維持の具体的方法

- 方針: **認可コード取得〜初回 token.json 作成までは今までどおり `main.py` フローを維持**する。
- 手順:
  1. 初回セットアップ/再認可は既存どおり `python main.py --auth-code <認可コード>` で実施。
  2. `web.py` 起動時は `token.json` の存在と `refresh_token` 形式のみチェック。
  3. 送信時は既存関数（`load_refresh_token -> refresh_access_token -> save_tokens`）を呼び、従来と同じ更新経路で token.json を更新してから本処理へ進む。
  4. リフレッシュ失敗時は処理を中断し、「既存の再認可コマンドを実行してね」と案内する。
- これで「トークン運用の責務は auth 層の既存実装を使う」を維持できる。

### 5) フォーム項目とバリデーション方針

- 対象月 (`target_month`)
  - 入力形式: `yyyy-mm`
  - バリデーション: `^\d{4}-\d{2}$` + 実在年月チェック
- 出勤何時 (`start_hour`)
  - 入力形式: 数値（0-23）
  - バリデーション: 整数、範囲内
- 退勤何時 (`end_hour`)
  - 入力形式: 数値（0-23）
  - バリデーション: 整数、範囲内、かつ `start_hour < end_hour`
- 従業員ID (`employee_id`)
  - 入力形式: 数値
  - バリデーション: `^\d+$` かつ `>= 1`
- 出社タグ (`include_attendance_tag`)
  - 入力形式: チェックボックス
  - デフォルト: OFF（既存 CLI の Enter=つけない と同等）
- ルール:
  - クライアント側（HTML）とサーバ側（Python）の**二重バリデーション**を行う。
  - 判定基準は `bulk_attendance.py` の既存仕様に合わせる。

### 6) 送信後のぐるぐるUIと待機メッセージ表示案

- 画面状態を 3 段階に分ける。
  - `idle`: 入力可能
  - `submitting`: ぐるぐる表示 + 送信ボタン無効化 + 二重送信防止
  - `done/error`: 結果表示
- `submitting` 時のメッセージ案:
  - `勤怠データを登録中です。月の日数ぶん API を順番に呼ぶので、少し待ってね。`
  - `この画面は閉じずにそのまま待機してください。`
- 任意で表示する補助情報:
  - 対象月
  - 対象従業員ID
  - 開始時刻（処理開始 timestamp）

### 7) セキュリティ/運用上の注意点

- Web サーバは `127.0.0.1` 利用前提で外部公開しない。
- `token.json` と API レスポンスをログへ生で出さない（アクセストークン/個人情報マスク）。
- 従業員ID指定機能は誤操作影響が大きいので、送信前確認（対象月・従業員ID・勤務時間）を必須化する。
- 同時実行を避けるため、単一実行ロック（例: 実行中フラグ）を設ける。
- freee API への呼び出し間隔（既存の待機秒）は維持し、レート制限リスクを上げない。

### 8) 実装に進む前にオーナーが決めるべきポイント

- 案A（同期実行）で進めるか、将来見越しで案B（非同期ジョブ）にするか。
- 利用ポート（例: `8000`）と起動コマンドの最終形（`-p` 運用か `ports` 定義追加か）。
- 従業員IDの操作許可範囲（誰でも任意ID指定可能にするか、許可リストで制限するか）。
- 送信後画面の粒度（完了/失敗だけでよいか、日次進捗まで出すか）。
- エラー時の再試行ポリシー（手動再送のみ or 自動リトライあり）。
