# 018 指定月の従業員ID指定勤怠登録機能設計

## 要望

既存機能「指定の月に自分の勤怠を詰め込む」を残したまま、新機能「指定の月に従業員ID指定で勤怠を詰め込む」を追加する。

## 設計方針（差分最小）

- 既存の `bulk_attendance` の月次処理ロジック（有給判定、日次ループ、勤怠 PUT、タグ PUT、サマリー出力）はそのまま再利用する。
- 差分は「対象 `employee_id` の決定方法」だけに寄せる。
- 既存メニュー「指定の月に自分の勤怠を詰め込む」は挙動を変えない。

## 詳細設計

### 1. 新機能のメニュー文言と識別子

- 追加する `action_id`: `bulk_attendance_by_employee_id`
- 追加する `menu_label`: `指定の月に従業員ID指定で勤怠を詰め込む`
- 登録先: `app/actions/registry.py`
- 既存の `action_id="bulk_attendance"` は変更しない（回帰防止）

### 2. `company_id` と `employee_id` の取得フロー

- `company_id` は `GET /hr/api/v1/users/me` から取得する。
- 取得対象は `companies[0].id` のみ。今回の新機能では `users/me` 側の `employee_id` は使わない。
- `employee_id` はメニュー選択後の対話入力（`input()`）から取得する。
- 処理順は次のとおり。
1. 対象月入力（既存）
2. 出退勤時刻入力（既存）
3. `users/me` で `company_id` 取得
4. `employee_id` を入力で取得
5. `attendance_tags` 取得以降は既存フローと同一

### 3. `employee_id` 入力バリデーションと案内文

- 入力プロンプト文言案:
`対象の従業員IDを入力してね (数字のみ。よくわからんかったら Ctrl + C でいったん終わって、やり直してね):`
- バリデーション:
  - 空文字はエラー
  - `^\d+$` 以外はエラー
  - `int(employee_id) >= 1` を必須
- エラーメッセージ案:
  - `[エラー] 従業員IDは空で入力できないよ`
  - `[エラー] 従業員IDは数字だけで入力してね: 'abc'`
  - `[エラー] 従業員IDは 1 以上の整数で入力してね: '0'`
- `Ctrl+C` は中断として扱う。案内文どおり、いったん終了してやり直せる導線にする。

### 4. 既存 `bulk_attendance` ロジック再利用方針（共通化点と分岐点）

- 共通化点（再利用する処理）:
  - `_parse_target_month`
  - `_parse_work_hours`
  - `_resolve_attendance_tag_id`
  - `_load_paid_holidays_by_date`
  - `_generate_dates`
  - `_process_date`
  - `_print_summary`
- 分岐点（新規追加する処理）:
  - `employee_id` の解決方法のみ
  - 既存: `_resolve_user_ids` で `company_id` と「自分の `employee_id`」を取得
  - 新規: `users/me` で `company_id` のみ取得 + `_parse_employee_id` で入力値採用
- 実装イメージ（最小差分）:
  - `app/actions/bulk_attendance.py` に共通実行関数（例: `_run_bulk_attendance(...)`）を切り出す
  - 既存 `handler` は従来どおり「自分の `employee_id`」で `_run_bulk_attendance(...)` を呼ぶ
  - 新規 `handler_by_employee_id` を追加し、入力 `employee_id` で `_run_bulk_attendance(...)` を呼ぶ

### 5. エラー処理方針

- 入力エラー（対象月、時刻、`employee_id`）:
  - 既存方針どおり終了コード `2`（`EXIT_CODE_MENU_ERROR`）
- `users/me` で `company_id` を取れない:
  - `ActionExecutionError` として終了コード `1`
- 対象 `employee_id` が不正/権限外で API 失敗:
  - 既存 `_print_api_error` 形式で日付・ステップ・status を出して即中断
  - 終了コード `1`
- 日次処理中エラー:
  - 現行どおり「失敗した日で中断」を維持
- `Ctrl+C`:
  - ユーザー中断（終了コード `130`）として扱う前提は維持

### 6. テスト観点

- メニュー/登録:
  - `registry` に `bulk_attendance_by_employee_id` が追加され、既存 `bulk_attendance` は残っている
- 入力バリデーション:
  - 正常: `"123"` を `int` 化できる
  - 異常: 空文字、英字混在、`0`、負数文字列でエラー終了
  - `Ctrl+C` 中断
- ID解決:
  - 新機能では `company_id` を `users/me` から取得し、`employee_id` は入力値を採用している
  - 既存機能は従来どおり `users/me` の `employee_id` を採用している
- 処理回帰:
  - 新機能で `employee_id` を渡したとき、`attendance_tags` 取得以降の処理結果（SKIP/OK/ERR ログ、サマリー）が既存と同じ規則で動く
- APIエラー:
  - `attendance_tags` / `work_records` / `attendance_tags PUT` の失敗時に既存どおり即中断する

- 実装時に迷わないよう、分岐点を `employee_id` 解決の 1 点に絞って進める。

## オーナー向け要約（実装）

- `app/actions/registry.py` に `bulk_attendance_by_employee_id` を追加し、メニュー文言を `指定の月に従業員ID指定で勤怠を詰め込む` として登録した。
- `app/actions/bulk_attendance.py` に `handler_by_employee_id` を追加し、既存処理は `_run_bulk_attendance` / `_execute_bulk_attendance` へ共通化して再利用する構成にした。
- 新規入力 `_parse_employee_id` を追加し、案内文に `よくわからんかったら Ctrl + C でいったん終わって、やり直してね` を含めた。
- `employee_id` バリデーション（空・数字以外・1未満）を実装し、入力不正時は既存方針どおり `EXIT_CODE_MENU_ERROR (2)` を返す導線にした。
- `company_id` は新旧どちらの導線でも `users/me` から取得する実装とし、既存「自分の勤怠」導線は従来の `employee_id` 解決を維持した。
- テスト `tests/test_bulk_attendance_paid_holidays.py` を更新し、以下を追加確認した。
  - 新メニュー `bulk_attendance_by_employee_id` の登録
  - `employee_id` 入力バリデーション
  - 新機能で `company_id` が `users/me`、`employee_id` が入力値で解決されること
