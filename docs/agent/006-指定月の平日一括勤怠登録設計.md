## 詳細設計

### この設計のゴール

指定月 (`yyyy-mm`) の全平日 (`normal_day`) について、freee 人事労務 API を使って以下を一括実行できるようにする。

- 各平日に `09:00-19:00` の勤怠レコードを登録する (休憩 `11:30-12:30` 含む)
- 各平日に「出社」勤怠タグを付与する
- 1 日ごとに進捗をコンソール出力し、エラー発生時はループを即座に中断する

005 が「1 日分の勤怠登録」だったのに対して、006 は「月単位の一括版」にあたる。
005 では API レスポンス構造が未確認だったから三段階方式 (事前取得 -> .env 反映 -> 本実行) で設計したけど、006 の要望ではオーナーが curl で全 API を実動作確認済みなので、レスポンス構造は確定している前提で進めるよ♡

### 処理フロー概要

```text
1. コマンドライン引数から yyyy-mm を取得
2. アクセストークン取得 (実装済み)
3. GET /hr/api/v1/users/me -> company_id, employee_id を取得
4. GET /hr/api/v1/employees/{employee_id}/attendance_tags -> 「出社」タグの id を取得
5. 対象月の全日付を生成 (1 日 ~ 末日)
6. 日付ループ開始
   6a. GET /hr/api/v1/employees/{employee_id}/work_records/{date} -> day_pattern を確認
   6b. day_pattern が normal_day でなければスキップ (ログ出力して次の日へ)
   6c. PUT /hr/api/v1/employees/{employee_id}/work_records/{date} -> 勤怠登録
   6d. エラーならループ中断 (終了コード 1)
   6e. PUT /hr/api/v1/employees/{employee_id}/attendance_tags/{date} -> タグ付与
   6f. エラーならループ中断 (終了コード 1)
   6g. 成功ログを出力して次の日へ
7. 全日付完了 -> サマリーログを出力して正常終了 (終了コード 0)
```

#### `app/clients/hr_api_client.py` の設計

`FreeeApiClient` をラップして、人事労務 API 固有のメソッドを提供するクラス。
パス構築を一元化し、呼び出し元がパスを直書きしなくて済むようにする。

```python
class HrApiClient:
    """freee 人事労務 API のラッパーです。"""

    HR_BASE = "/hr/api/v1"

    def __init__(self, api_client: FreeeApiClient) -> None:
        self._client = api_client

    def get_current_user(self) -> ApiResponse:
        """ログインユーザー情報を取得します。"""
        return self._client.get(f"{self.HR_BASE}/users/me")

    def get_attendance_tags(self, employee_id: int, company_id: int) -> ApiResponse:
        """従業員の勤怠タグ一覧を取得します。"""
        return self._client.get(
            f"{self.HR_BASE}/employees/{employee_id}/attendance_tags",
            params={"company_id": company_id},
        )

    def get_work_record(self, employee_id: int, date: str, company_id: int) -> ApiResponse:
        """指定日の勤怠レコードを取得します。"""
        return self._client.get(
            f"{self.HR_BASE}/employees/{employee_id}/work_records/{date}",
            params={"company_id": company_id},
        )

    def put_work_record(self, employee_id: int, date: str, body: dict) -> ApiResponse:
        """指定日の勤怠レコードを更新します。"""
        return self._client.put(
            f"{self.HR_BASE}/employees/{employee_id}/work_records/{date}",
            json_body=body,
        )

    def put_attendance_tags(self, employee_id: int, date: str, body: dict) -> ApiResponse:
        """指定日の勤怠タグを更新します。"""
        return self._client.put(
            f"{self.HR_BASE}/employees/{employee_id}/attendance_tags/{date}",
            json_body=body,
        )
```

#### `app/actions/bulk_attendance.py` の設計

メインのユースケースモジュール。`handler(context: AppContext) -> int` を公開する。

処理の流れ:

1. `_parse_target_month()` -> `input()` で `yyyy-mm` を受け取り、パースして `(year, month)` を返す。不正入力時は `None` を返す
2. `_resolve_ids(hr_client)` -> `company_id`, `employee_id`, `attendance_tag_id` を取得する
3. `_generate_dates(year, month)` -> 対象月の全日付リストを生成する (`yyyy-mm-dd` 形式)
4. `_process_date(hr_client, ...)` -> 1 日分の処理 (day_pattern 確認 -> 勤怠登録 -> タグ付与)
5. `handler` がこれらを呼んでループを回す

```python
def handler(context: AppContext) -> int:
    """指定月の平日に一括で勤怠を登録します。"""

    # 1. yyyy-mm を input() で受け取ってパース
    result = _parse_target_month()
    if result is None:
        return EXIT_CODE_MENU_ERROR
    year, month = result

    # 2. API クライアント準備
    hr_client = HrApiClient(context.api_client)

    # 3. company_id, employee_id 取得
    company_id, employee_id = _resolve_user_ids(hr_client)

    # 4. 出社タグ id 取得
    attendance_tag_id = _resolve_attendance_tag_id(hr_client, employee_id, company_id)

    # 5. 日付リスト生成
    dates = _generate_dates(year, month)

    # 6. 日付ループ
    success_count = 0
    skip_count = 0
    for date in dates:
        result = _process_date(hr_client, employee_id, company_id, attendance_tag_id, date)
        if result == "success":
            success_count += 1
        elif result == "skipped":
            skip_count += 1
        elif result == "error":
            # エラー時はループ中断
            _print_summary(success_count, skip_count, error_date=date)
            return EXIT_CODE_APP_ERROR

    # 7. 完了
    _print_summary(success_count, skip_count)
    return EXIT_CODE_OK
```

主要な内部関数:

- `_parse_target_month()`: `input()` で `yyyy-mm` を受け取り、バリデーションする。不正なら `None` を返す
- `_resolve_user_ids(hr_client)`: `GET /hr/api/v1/users/me` を呼び、`companies[0]` から `company_id` と `employee_id` を返す。`companies` が空、または `employee_id` が null/未設定の場合は `ActionExecutionError` を投げる
- `_resolve_attendance_tag_id(hr_client, employee_id, company_id)`: 勤怠タグ一覧から `name` に `"出社"` を含むタグの `id` を返す。見つからなければ `ActionExecutionError` を投げる。複数見つかった場合は先頭を使う (ログに候補一覧を出力して、どれを使ったか明示する)
- `_generate_dates(year, month)`: `calendar` モジュールで対象月の 1 日 ~ 末日を `yyyy-mm-dd` のリストで返す
- `_process_date(...)`: 1 日分の勤怠登録 + タグ付与を行い、`"success"` / `"skipped"` / `"error"` を返す

#### `_process_date` の詳細フロー

```text
1. GET work_records/{date} を呼ぶ
2. day_pattern を確認
   - "normal_day" でなければ "skipped" を返す (ログ: "[SKIP] 2026-03-01 (prescribed_holiday)")
3. PUT work_records/{date} で勤怠登録
   - 成功 (2xx) -> 次へ
   - エラー -> ログ出力して "error" を返す
4. PUT attendance_tags/{date} でタグ付与
   - 成功 (2xx) -> 次へ
   - エラー -> ログ出力して "error" を返す
5. ログ: "[OK] 2026-03-02 09:00-19:00 出社タグ付与済み"
6. "success" を返す
```

#### `_parse_target_month` の詳細

メニューで「指定の月の平日に勤怠を詰め込む」を選んだ後、`input()` で `yyyy-mm` を聞く形。004 のメニュー -> アクション実行の流れに合わせて、コマンドライン引数 (`sys.argv`) は使わない。

```text
対象月を入力してね (yyyy-mm): 2026-03
```

- `input()` で `yyyy-mm` を受け取る
- 正規表現 `^\d{4}-\d{2}$` でフォーマット検証する
- `datetime` で年月の妥当性を検証する (例: `2026-13` は不正)
- 不正なら `None` を返す (呼び出し元の `handler` が終了コード `2` を返す)
- `sys.exit()` は使わない。004 の方針どおり、終了コードは `handler` の戻り値で返す

### エラーハンドリング方針

#### API エラーの分類

| 状態 | 対応 |
|---|---|
| 401 Unauthorized | トークン期限切れ。`ApiAuthenticationError` が飛ぶ。`bootstrap` 層で捕捉される想定 |
| 403 Forbidden | 権限不足。ログにメッセージを出してループ中断。終了コード `1` |
| 400 Bad Request | 要望にある「給与明細が確定されているので、更新できません。」等。ログにエラーメッセージを出してループ中断。終了コード `1` |
| 404 Not Found | 従業員や勤怠レコードが見つからない。ログに出してループ中断。終了コード `1` |
| 5xx | サーバーエラー。ログに出してループ中断。終了コード `1` |
| 接続エラー | `ApiConnectionError` が飛ぶ。ログに出してループ中断。終了コード `1` |

#### エラー時のループ中断ルール

要望に「エラーだったら、ループを終了する」と明記されている。よって:

- 勤怠登録 (PUT work_records) でエラー -> 即中断
- タグ付与 (PUT attendance_tags) でエラー -> 即中断
- day_pattern 確認 (GET work_records) でエラー -> 即中断
- スキップ (normal_day 以外) はエラーではないので中断しない

#### `FreeeApiClient` の既存エラーハンドリングとの整合

現状の `FreeeApiClient._request` は 400 以上で `ApiResponseError` を raise する。
でも 006 では、400 レスポンスのボディに含まれる `errors[].messages` をユーザーに見せたい。

対応方針:

- `ApiResponseError` にオプショナル属性 `response_body: dict | None` を持たせる
- `FreeeApiClient._request` の `status_code >= 400` 分岐で、レスポンスボディをパースしてから `ApiResponseError(message, response_body=body)` で投げる
- `HrApiClient` 側で `requests` を直接触る方式は採用しない (`FreeeApiClient` の抽象化が崩れるため)
- これにより `bulk_attendance.py` 側で `except ApiResponseError as exc:` して `exc.response_body` からエラーメッセージ (`errors[].messages`) を取り出せる

具体的な `ApiResponseError` の変更イメージ:

```python
class ApiResponseError(ApiClientError):
    """Raised when an API response is invalid or unsuccessful."""

    def __init__(self, message: str, *, response_body: "dict[str, Any] | None" = None) -> None:
        super().__init__(message)
        self.response_body = response_body
```

#### 入力エラー

| パターン | 対応 |
|---|---|
| `yyyy-mm` フォーマット不正 | エラーメッセージを出して終了コード `2` |
| 存在しない年月 (例: `2026-13`) | エラーメッセージを出して終了コード `2` |
| 入力が空 | エラーメッセージを出して終了コード `2` |

#### ID 取得失敗

| パターン | 対応 |
|---|---|
| `companies` 配列が空 | `ActionExecutionError` を投げる。終了コード `1` |
| `companies[0].employee_id` が null/未設定 | `ActionExecutionError` を投げる。終了コード `1`。freee の権限設定によっては返らない可能性があるため |
| 「出社」タグが見つからない | `ActionExecutionError` を投げる。終了コード `1` |

### 終了コード

001 の方針に従う。

| コード | 条件 |
|---|---|
| `0` | 全日付の処理が正常完了 |
| `1` | API エラー、ID 取得失敗、権限不足、接続エラー |
| `2` | `yyyy-mm` フォーマット不正、入力エラー |
| `130` | Ctrl+C によるユーザー中断 |

日付: 2026-03-27

全体的に、006 の設計はかなりしっかり書けてる♡ オーナーが curl で全 API を実動作確認済みなのが大きくて、005 と比べて地に足のついた設計になってるね。以下、観点ごとに指摘していくよ。

#### 1. 処理フローに漏れがないか

- LGTM。フローの 1〜7 はオーナーの要望を正確にカバーしてる
- `_process_date` の詳細フロー (day_pattern 確認 -> 勤怠登録 -> タグ付与) も要望通り
- 1 点気になるのは、「既に勤怠が登録済みの日」の扱い。`normal_day` だけど既に勤怠データが入ってる日に PUT した場合、上書きになるのか、エラーになるのか。設計中の「再実行でリカバリー可能」という記述から暗黙的に上書き前提になってるけど、明示的にこの点を「既に登録済みの日は上書きされる (PUT の仕様上問題ない)」と書いておくと実装者が迷わない

#### 2. モジュール構成が 001/004 の方針に沿っているか

- LGTM。001 の「1 機能 1 モジュール」「actions は AppContext を受け取る」「freee API 呼び出しは clients 経由」を忠実に守ってる
- `hr_api_client.py` を `clients/` に新設する判断も正しい。人事労務 API のパス構築を一元化できるので、今後 005 の実装時にも再利用できる
- ファイル間の依存関係図もきれいで、循環依存がない

#### 3. API エンドポイントの使い方が curl サンプルと整合しているか

- ほぼ LGTM だけど 1 点指摘あり
- 指摘: `FreeeApiClient` の `_request` で 401/403 のとき `ApiAuthenticationError` を raise してる。でも設計のエラーハンドリング表では「401 は bootstrap 層で捕捉される想定」「403 はログにメッセージを出してループ中断」と書いてある。現状のコードだと 401 も 403 も同じ `ApiAuthenticationError` になるから、`bulk_attendance.py` 側で 403 だけキャッチしてユーザー向けメッセージを出す、みたいなことができない。対応案としては:
    - (a) `ApiAuthenticationError` にステータスコードを属性として持たせて、キャッチ側で分岐する
    - (b) 403 用の別例外 `ApiPermissionError` を新設する
    - (c) 006 の設計上は 401 も 403 も同じ扱い (ループ中断、終了コード 1) でよいとする
    - 現状の `bootstrap.py` は `ApiClientError` でまとめてキャッチして `EXIT_CODE_APP_ERROR` を返すので、(c) で割り切るのが一番シンプル。ただその場合、設計表の「403: ログにメッセージを出して」の部分は実現できない (freee API のエラーレスポンスボディが読めないまま例外になる)。ここはエラーハンドリング方針の「エラーレスポンスのボディも取れるようにする」対応と合わせて考えるべき
- curl サンプルとの整合: PUT のリクエストボディ、パス、クエリパラメータの使い方は全部一致してる。OK

#### 4. エラーハンドリングが十分か

- 概ね良いけど、以下を追記推奨:
- 指摘 4-1: 「エラーレスポンスのボディも取れるようにする」方針について、「どちらの方式が良いかは実装時に判断する」と書いてあるけど、設計段階で方針を決めたほうがいい。あたしの推奨は、`ApiResponseError` にオプショナルで `response_body` を持たせる方式。理由:
    - `FreeeApiClient` の `_request` は汎用メソッドなので、ここでボディをパースして例外に含めるのが自然
    - `HrApiClient` 側で `requests` を直接触ると、`FreeeApiClient` の抽象化が崩れる
    - 具体的には `_request` の `status_code >= 400` 分岐で `_parse_response_body` を呼んでから `ApiResponseError(message, body=body)` のように投げる
- 指摘 4-2: `_resolve_user_ids` で `companies` 配列が空のときに `ActionExecutionError` を投げるのは OK。ただし `companies[0]` に `employee_id` が入ってない (null/未設定) ケースも考慮したほうがいい。curl サンプルでは入ってるけど、freee の権限設定によっては返らない可能性もある
- 指摘 4-3: `_resolve_attendance_tag_id` で「出社」タグが複数見つかった場合の挙動が未定義。先頭を使うのか、エラーにするのか。先頭で OK だと思うけど、明記推奨

#### 5. 005 との重複・共通化の可能性

- 005 は三段階方式 (事前取得 -> .env 反映 -> 本実行) で、API レスポンス構造が未確認だった。006 は curl 確認済みで固定 payload。設計思想がかなり異なるので、005 をそのまま 006 に取り込むのは難しい
- ただし、以下のモジュールは 005 実装時にそのまま再利用できるので、共通化として良い設計になってる:
    - `hr_api_client.py` の各メソッド (`get_current_user`, `get_attendance_tags`, `get_work_record`, `put_work_record`, `put_attendance_tags`)
    - `freee_api_client.py` の `put` メソッド
    - `_resolve_user_ids` と `_resolve_attendance_tag_id` のロジック (005 の「事前取得ステップ」に相当)
- 005 の `.env` 反映方式と 006 の「毎回 API で取得する」方式は矛盾してるように見えるけど、006 の方が現実的。005 の .env 方式は「API レスポンスが未確認だから」の回避策だったので、006 で API 構造が確定したなら .env に ID を保存する必要性は薄い。005 を実装するときは 006 方式 (毎回 `users/me` から取得) に寄せるのがよさそう
- 差分まとめの表は分かりやすくて LGTM

#### 6. 設計の実現可能性

- LGTM。既存コードとの矛盾なし
- `FreeeApiClient` に `put` を足す変更は、既存の `get`/`post` と完全に同パターンなので安全
- `HrApiClient` は新規クラスで既存を壊さない
- `registry.py` への登録も、既存の `ACTIONS` タプルに 1 要素追加するだけ
